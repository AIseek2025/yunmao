# 02-高清低延迟直播架构

## 1. 直播体验目标

yunmao 的直播体验必须支撑“高清晰度、低卡顿、低延迟、可规模化分发”四个目标：

| 指标 | MVP 目标 | 成长期目标 |
| --- | --- | --- |
| 首帧时间 P95 | ≤ 1.5 秒 | ≤ 1.0 秒 |
| 播放卡顿率 | ≤ 3% | ≤ 1.5% |
| 投喂点击到画面可感知 P95 | ≤ 5 秒 | ≤ 3 秒 |
| 端到端直播延迟 | HLS ≤ 8 秒，低延迟房间 ≤ 3 秒 | LL-HLS/HTTP-FLV ≤ 3 秒，WebRTC ≤ 1 秒 |
| 清晰度 | 1080p 主档，720p/480p 自适应 | 1080p 主档，重点房间 2K/4K 试点 |

## 2. 流媒体链路

```text
摄像头/边缘盒子
  ├─ RTSP/USB 摄像头采集
  ├─ 本地弱网缓存、断线重连、画面健康检测
  └─ SRT/RTMP/WebRTC 推流

直播接入层
  ├─ 云直播接入或 Rust 媒体边缘接入
  ├─ 鉴权、防盗推、推流质量检测
  └─ 多码率转码、录制、关键帧截图

分发层
  ├─ CDN HLS/LL-HLS
  ├─ HTTP-FLV/WS-FLV 低延迟分发
  └─ WebRTC SFU 强互动房间

播放端
  ├─ 自动选择协议和码率
  ├─ Web/H5 与 App 播放体验一致性
  ├─ 上报 QoE
  └─ 投喂事件与画面时间轴对齐
```

## 3. 推流端设计

### 设备组合

- MVP 推荐：网络摄像头或 USB 摄像头 + 边缘盒子/树莓派/小主机 + 独立投喂机。
- 救助站/猫咖推荐：PoE 摄像头 + 局域网边缘代理 + 有线网络优先。
- 家庭用户推荐：认证摄像头 + 认证投喂机，降低配置成本。

### 推流参数

| 档位 | 分辨率 | 帧率 | 建议码率 | 用途 |
| --- | --- | --- | --- | --- |
| 超清 | 1080p | 25/30 fps | 3-6 Mbps | 默认高清观看 |
| 高清 | 720p | 25/30 fps | 1.5-3 Mbps | 移动网络或多人观看 |
| 流畅 | 480p | 20/25 fps | 600 Kbps-1.2 Mbps | 弱网兜底 |

编码建议：

- H.264 作为主编码，保证 Web 与 App 兼容；H.265/AV1 可作为未来成本优化。
- GOP 1-2 秒，低延迟房间控制在 1 秒左右。
- 禁止过度 B 帧，避免投喂反馈延迟被播放器缓冲放大。
- 边缘盒子需要定期上报丢帧、编码耗时、上行带宽、推流重连次数。

## 4. 播放协议策略

| 协议 | 延迟 | 优点 | 缺点 | 使用场景 |
| --- | --- | --- | --- | --- |
| HLS | 5-12 秒 | 兼容性最好、CDN 成本低 | 延迟较高 | 首页预览、超热门围观房间、弱互动 |
| LL-HLS | 2-5 秒 | 兼容较好、延迟可控 | 云厂商/播放器配置要求高 | Web 与 App 主播放方案之一 |
| HTTP-FLV/WS-FLV | 1-3 秒 | 低延迟、实现成熟 | iOS 与移动端兼容需处理 | 桌面 Web 低延迟房间、Android 试点 |
| WebRTC | 300ms-1 秒 | 强互动、低延迟 | 成本高、调度复杂，移动端耗电和权限复杂 | 投喂强反馈、连线、重点房间 |

MVP 默认策略：

- 移动 H5：优先 HLS/LL-HLS，支持自动降级。
- 桌面 Web：优先 HTTP-FLV/LL-HLS。
- iOS/Android App：优先 HLS/LL-HLS 或云厂商低延迟播放器 SDK；仅对重点强互动房间启用 WebRTC。
- “投喂演示房间”或强互动活动：使用 WebRTC 或 HTTP-FLV。

App 首期必须覆盖：

- 弱网策略：检测蜂窝网络、丢包和频繁切网，自动降到 720p/480p，并明确提示用户。
- 前后台策略：进入后台后默认暂停直播或降级为音频/通知，不承诺首期后台持续视频播放；恢复前台需快速续播。
- 权限策略：通知权限、网络权限、麦克风/相机权限只在相关功能触发时申请，首期观看和投喂不依赖麦克风/相机权限。
- 推送联动：开播、可投喂、投喂结果、设备异常等通知必须能深链回房间页。

## 5. 低延迟投喂体验设计

投喂体验由两个链路组成：

- 控制链路：用户点击 → Go 投喂服务 → 队列 → MQTT → 投喂机回执。
- 画面链路：摄像头采集 → 推流 → 分发 → 播放器缓冲 → 用户看到猫粮掉落。

为了让用户感知“我喂到了”，需要：

- 前端在指令被服务端接受后立即显示“已接收/排队中”，不能提前展示“已出粮”。
- 设备回执成功后显示“出粮成功”，并在直播画面叠加时间同步提示。
- 低延迟房间将播放器缓冲控制在 1-2 秒，投喂期间临时降低缓冲策略。
- App 端投喂过程中若发生后台切换或网络切换，必须保留请求状态并通过站内消息/推送补齐结果。
- 设备端可在出粮瞬间触发灯光或提示音，便于画面识别。
- AI/边缘截图服务在设备回执后抓取 5-10 秒窗口，生成投喂回放。

## 6. 高画质保障

### 采集侧

- 摄像头至少支持 1080p、低照度、自动曝光锁定。
- 投喂区域必须有固定光源，避免夜间画面噪点导致 AI 误判。
- 摄像头角度要覆盖投喂口、碗和猫咪进入路径。

### 编码侧

- 使用固定码率上限 + 自适应码率控制，避免弱网上行导致严重花屏。
- 对重点房间启用云端或边缘转码，输出多码率自适应播放列表。
- 关键房间可试点 AI 超分/降噪，但不得影响主链路稳定性。

### 播放侧

- 播放器支持自适应码率、手动切换清晰度、失败重试和协议降级。
- 首屏先播放低码率，稳定后升档到高清。
- 房间页展示“直播下线/维护/设备离线/投喂关闭”的明确状态，避免用户误判。
- Web 与 App 使用同一套播放配置接口，但 App 可根据系统、机型、电量、网络类型选择更保守的默认码率。

## 7. 大规模扩展方案

### 百房间到万房间

- 使用云直播 + CDN 作为主干。
- Go 房间服务维护推拉流地址、鉴权 token、房间状态。
- 重点房间开启录制、转码、截图；普通房间按需开启。

### 万房间到百万设备

- 引入区域级 Rust 媒体边缘网关，靠近设备接入点。
- 边缘网关负责推流健康检测、协议转换、弱网重连、关键帧抽取。
- 中心 Go 调度服务只处理元数据和策略，不承载媒体包。
- 热门房间自动升配 CDN 资源，冷门房间按需拉流，降低成本。

### 数千万用户观看

- 直播分发主要依赖 CDN 多区域节点。
- 房间热度驱动预热和多码率缓存。
- API 侧只承接轻量业务请求，并通过多级缓存提供房间数据。
- 实时互动分层：热门房间弹幕采样/分片，小房间全量广播。

## 8. 监控指标

必须按房间、地区、协议、清晰度维度监控：

- 推流在线率、推流重连次数、上行码率、丢帧率。
- 播放成功率、首帧时间、卡顿次数、卡顿总时长、播放失败码。
- App 前后台切换续播成功率、弱网降级次数、推送深链回房间成功率。
- CDN 回源率、边缘命中率、带宽峰值、单房间成本。
- 投喂回执到用户画面可见的延迟估算。
- AI 检测到猫咪/碗/出粮口的可见率。

## 9. 降级策略

- 直播源故障：房间进入“直播维护中”，保留聊天和历史精彩瞬间。
- CDN 或协议异常：WebRTC/FLV/HLS 自动切换，最终兜底到 HLS；App 端优先兜底到系统原生可稳定播放的 HLS。
- 带宽成本异常：热门房间限制最高码率，非会员/游客默认 720p。
- 高峰投喂：投喂排队、聚合投喂、关闭非核心特效。
- AI 处理积压：暂停短视频生成，只保留设备回执和核心截图。

## 10. 推流端编码管线

```text
摄像头 (RTSP/USB)
  → 边缘盒子采集 (V4L2 / RTSP 客户端)
  → 预处理 (去抖动 / 自动曝光锁定 / 时间戳同步)
  → 编码 (H.264 baseline/high, GOP=1-2s, B 帧=0-1)
  → 自适应码率控制 (CBR + 峰值上限)
  → 多档转封装 (源档/720/480)
  → 推流 (SRT 优先, RTMP 兜底; WebRTC 用于强互动)
  → 直播接入 (云直播 / Rust media-edge)
```

关键工程要求：

- **时间戳一致**：摄像头 NTP/PTP 同步，推流端在 SEI 中写入设备时间戳与 `device_id`，便于投喂事件与画面对齐。
- **GOP 与延迟取舍**：MVP GOP 1–2s 兼顾 CDN 缓存与延迟；强互动房间 GOP 1s，并禁用 B 帧。
- **码率自适应**：上行带宽不足时，先降码率（CBR 上限下调）→ 再降帧率 → 最后降分辨率，避免分辨率频繁跳变带来观感断层。
- **断线重连**：SRT/RTMP 必须支持 5s 内自动重连 + 关键帧重发；连续 3 次失败上报告警。
- **本地缓存**：边缘盒子可缓存最近 30s 的关键帧，断流恢复后可以拼接“无缝过桥”到 CDN。

## 11. ABR 与码率梯度

| 档位代号 | 分辨率 | 码率 | 帧率 | 适用 |
| --- | --- | --- | --- | --- |
| `src` | 源 1080p+ | 4–8 Mbps | 25/30 | 录制 / 高级订阅 |
| `hd1080` | 1080p | 3–4 Mbps | 25/30 | 桌面默认 / Wi-Fi |
| `hd720` | 720p | 1.5–2 Mbps | 25/30 | 移动 4G/5G 默认 |
| `sd480` | 480p | 600–1000 Kbps | 20/25 | 弱网兜底 |
| `audio` | — | 64–96 Kbps | — | 极弱网 / 后台 |

策略：

- 首屏先用 720p 起播 → 稳定 5s 后切换到目标档；首帧成本和卡顿率优于直接最高档。
- 客户端 ABR 决策窗口：3 个 chunk（约 6s）观察 → 升档需连续 2 次良好；降档遇 1 次缓冲耗尽立即下调。
- SVC（可分层编码）在 WebRTC 房间试点，主流房间继续走多档转码，避免边缘 SVC 解码兼容问题。
- 服务端可以对一类用户（如游客、非会员）强制限定最高档，控制成本（参见 10 章商业化策略）。

## 12. QoE 指标模型

播放端必须按以下字段上报每个会话：

- 维度：`room_id`、`user_id`、`client_type`、`os`、`os_version`、`app_version`、`network_type`、`isp`、`region`、`protocol`、`profile`、`cdn_provider`、`edge_node`
- 指标：`time_to_first_frame_ms`、`startup_failure_code`、`stall_count`、`stall_total_ms`、`avg_bitrate_bps`、`profile_switch_count`、`protocol_fallback_count`、`session_duration_ms`、`exit_reason`、`bg_resume_success`、`feed_overlay_delay_ms`
- 上报：会话开始时 1 次握手 + 每 30s 心跳 + 每次关键事件（首帧、卡顿、切档、失败、退出）
- 隐私：`user_id` 用一次性会话 ID 投递，原始 ID 在服务端关联，禁止把 `user_id` 与 IP 同时落 ClickHouse

服务端聚合视图（参见 06 第 2 节）：

- 全局首帧 P50/P95、卡顿率（卡顿时长 / 播放时长）、播放成功率
- 按机型/系统/网络/CDN/edge 多维下钻
- “投喂叠加延迟” = 设备回执时刻 → 用户客户端画面叠加时刻；超 SLA 自动告警

## 13. 多 CDN 与边缘演进

| 阶段 | 方案 | 说明 |
| --- | --- | --- |
| MVP | 单 CDN（云厂商直播 CDN） | 接入快、可用度高，调度由云厂商负责 |
| 成长期 | 双 CDN 主备 + 客户端 fallback 列表 | 服务端按可用度/单房间成本动态选主；客户端拿到候选 URL 列表，失败后切备 |
| 规模期 | 多 CDN + 自建 Rust 媒体边缘 PoP | 自有边缘负责热门房间和近源缓存；CDN 负责长尾覆盖；调度服务统一编排 |

调度策略：

- **客户端粒度**：按地区 + 网络 + 历史 QoE 选默认 CDN。
- **房间粒度**：热门房间预热到多 CDN；冷门按需。
- **失败回退**：3 次连续失败或卡顿率超过阈值 → 客户端自动切下一 CDN，并把事件上报。

自建边缘节点 PoP 设计（规模期）：

- 节点形态：Rust + libsrt + 自研 HTTP-FLV / LL-HLS 模块；可选 SVC/Simulcast。
- 路由：DNS + 实时调度（基于探测/容量），客户端 SDK 内置 fallback。
- 边缘缓存：HLS 切片缓存 + 关键帧预热；命中率目标 ≥ 95%。
- 边缘 AI 旁路：在节点上做轻量截图/检测，结果走 Kafka 回中心。

## 14. 大规模扩展数学模型（粗算）

以规模期 100 万同时在线、平均 1.5 Mbps 计算：

- 总出流量 ≈ 100 万 × 1.5 Mbps ≈ 1.5 Tbps
- 单 CDN 节点承载 200 Gbps，需 ≈ 7–10 个节点活跃
- 中心源站只承载推流和回源：单房间 1 路推流，回源率目标 ≤ 5%，源站出口 ≈ 75 Gbps，需多源协同
- 推流并发：数十万房间在线，按 4 Mbps 上行 → 中心媒体接入需要数十至上百台 Rust 边缘节点
- API 侧投喂峰值 5 万 QPS：网关 + 投喂服务无状态横向扩展，Redis 冷却 Cluster 至少 16–32 分片避免热点

以上仅为预算口径，实际数值由 05 第 4/5 节压测确定。

## 15. 抗弱网与降级补偿

| 网络情况 | 客户端策略 | 服务端策略 |
| --- | --- | --- |
| Wi-Fi/5G 稳定 | 默认 720/1080p，LL-HLS/HTTP-FLV | 正常 |
| 4G 一般 | 默认 720p，缓冲拉长 1s | 主推 LL-HLS |
| 弱 4G / 高丢包 | 降到 480p，禁用 WebRTC | 通过下发更短分片，鼓励 ABR 快收敛 |
| 弱信号边缘 | 切音频或暂停视频，保留文字状态 | 暂停媒体连接、保留长连接收事件 |
| 推流端弱网 | 边缘盒子降码率、丢非关键帧 | 触发录制断点续传、记录缺口 |
| App 切后台 | 释放解码器，保留长连接 + 推送 | 不再下发媒体数据，事件继续 |

补偿：

- 因网络波动错过的投喂结果，必须通过站内消息/推送补齐（参见 04 第 4 节）。
- 因弱网降到极低画质期间，不应记入个人“连续观看时长”等成就，避免误判。

## 16. 主播侧体验

- 提供推流诊断工具：在猫主人/机构端 App/Web 显示上行码率、丢帧、断线重连次数。
- 推流授权 token 短期有效（≤ 30 分钟），过期自动重签，避免推流地址被分享导致盗推。
- 救助站/猫咖建议有线网络优先；提供 4G/5G 备份模块作为兜底。
