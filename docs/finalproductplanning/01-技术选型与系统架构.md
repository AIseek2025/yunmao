# 01-技术选型与系统架构

## 1. 架构目标

yunmao 的架构必须同时满足三个阶段：

- MVP：少量房间、少量设备，Web/H5 与 iOS/Android App 同步交付核心链路，依赖云直播/CDN 快速验证。
- 成长期：数千到数万房间，百万级日活，弹幕/投喂/排行榜高峰明显。
- 规模期：数百万直播或投喂设备、数千万用户，媒体流量主要由边缘和 CDN 承载，控制面与数据面可水平扩展。

核心原则：

- 媒体流、控制流、业务流、数据分析流彻底解耦。
- 直播播放链路不依赖业务服务可用性；业务故障时至少保留观看。
- 投喂指令必须幂等、可审计、可回滚运营状态，不能被前端重复点击直接放大到设备。
- 从第一天定义 Rust + Go 的边界，避免后续重写全部系统。

| --- | --- | --- |

## 3. Rust + Go 分工边界

### Go 负责控制面与业务编排

Go 适合高并发但业务变化快的服务：

- 用户服务：手机号/微信登录、账号、权限、风控标签、多端会话。
- 房间服务：房间资料、多端直播地址、权限、冷却配置、房间状态。
- 投喂服务：资格校验、冷却、队列入站、幂等、结果聚合。
- 设备控制面：设备绑定、证书发放、固件版本、配置下发。
- 支付与订单：微信支付、支付宝、退款、分账、对账。
- 社交与增长：弹幕审核策略、评论、关注、分享、邀请、任务。
- 通知服务：站内消息、App 推送 token、开播/可投喂/投喂结果通知、退订与频控。
- 运营后台：房间审核、内容审核、机构管理、数据看板。

选择 Go 的原因：

- 业务 SDK 生态成熟，支付、短信、云厂商、数据库支持好。
- goroutine 足以支撑大规模 API、WebSocket 控制消息和后台任务。
- 团队招聘和维护成本低，适合作为平台主语言。

### Rust 负责高性能数据面和低延迟边界

Rust 优先用于 GC 延迟毛刺敏感、二进制数据密集、边缘部署密集的场景：

- 媒体边缘网关：RTMP/SRT/RTSP 接入、WebRTC SFU/转发、LL-HLS 切片、HTTP-FLV 转发。
- 协议转换：摄像头 RTSP → SRT/RTMP/WebRTC，边缘节点本地重连和降级。
- 高密度实时网关：单机高连接数 WebSocket/QUIC 网关，弹幕广播和在线状态。
- 视频处理旁路：FFmpeg/GStreamer 管线编排、关键帧截图、短视频切片。
- AI 推理适配：TensorRT/ONNX Runtime/libtorch 的安全封装，输出事件给 Go。
- 设备边缘代理：救助站/猫咖本地网关，弱网缓存、指令本地排队、设备健康探测。

选择 Rust 的原因：

- 无 GC，适合媒体包转发、零拷贝缓冲区、低尾延迟。
- 内存安全，适合长期运行在边缘节点和设备网关。
- 与 C/C++ 媒体/推理库集成可控。

### 边界契约

- Go 与 Rust 服务之间统一使用 gRPC + Protobuf；事件流使用 Kafka/RocketMQ。
- Rust 不直接写业务主库，只上报媒体状态、设备状态、AI 事件、QoE 指标。
- Go 不直接处理重型媒体包，只保存媒体元数据、调度策略和权限。
- 所有跨语言接口必须版本化，禁止临时 JSON 字段长期漂移。

## 4. 总体架构

```text
客户端(Web/H5/App/小程序)
  ├─ 业务 API: HTTPS/gRPC-Web
  ├─ 实时互动: WebSocket/QUIC
  └─ 直播播放: HLS/LL-HLS/HTTP-FLV/WebRTC

接入层
  ├─ CDN/WAF/DDoS
  ├─ API Gateway(鉴权、限流、灰度)
  └─ Live Edge/CDN(直播分发)

Go 控制面
  ├─ 用户/房间/投喂/支付/运营后台
  ├─ 设备控制面/证书/配置
  └─ 排行榜/分享/多端通知/风控

Rust 数据面
  ├─ 媒体边缘网关
  ├─ 实时消息网关
  ├─ 视频切片/截图/AI 推理旁路
  └─ 救助站本地边缘代理

中间件与数据
  ├─ Redis Cluster
  ├─ Kafka/RocketMQ
  ├─ MySQL/PostgreSQL/TiDB
  ├─ ClickHouse/Prometheus/Loki
  └─ OSS/S3 + CDN

IoT 设备层
  ├─ EMQX/HiveMQ MQTT Broker
  ├─ 摄像头/边缘盒子/投喂机/称重传感器
  └─ 设备证书、ACL、OTA、遥测
```

## 5. 高并发策略

### 直播并发

- 直播大流量走 CDN 和媒体边缘，业务 API 只返回播放地址和鉴权 token。
- 热门房间预热多码率流，避免突发回源。
- 按房间热度和客户端能力动态选择协议：普通房间 HLS/LL-HLS，桌面低延迟房间 HTTP-FLV/LL-HLS，App 优先 HLS/LL-HLS 或云厂商低延迟 SDK，强互动房间 WebRTC。
- 播放质量指标实时回传：首帧时间、卡顿率、码率切换、失败原因、App 前后台切换、弱网降级和系统权限状态。

### 投喂并发

- 前端防重复点击只是体验优化，真实防线在 Go 投喂服务和 Redis 原子状态机。
- 每次投喂生成 `feed_request_id`，一条请求只能驱动一次设备指令。
- 请求先校验用户、房间、冷却、动物福利上限，再进入消息队列。
- 同一设备消费者按设备维度串行处理，避免并发指令撞车。
- 高峰时可将多个用户的“助力投喂”聚合成一次真实出粮，并记录贡献人。

### 实时互动

- 弹幕写入与广播分离：写入服务负责审核和落事件，Rust/Go 实时网关负责广播。
- 热门房间按分片广播，低优先级弹幕采样展示。
- 弹幕、点赞、在线人数允许降级；投喂和直播优先级最高。

## 6. 演进路径

| 阶段 | 媒体方案 | Rust 投入 | Go 投入 |
| --- | --- | --- | --- |
| MVP | 云直播 + CDN，少量 WebRTC/HTTP-FLV；Web 与 App 共享播放鉴权和 QoE 上报 | 可先只做轻量截图/转码 worker，或暂不启用 | 完成账号、房间、投喂、通知、支付预留、运营后台等多端业务闭环 |
| 成长期 | 云直播 + 自建边缘代理混合 | 上线媒体探测、截图、实时网关 | 服务拆分、支付、机构后台、风控、App 增强通知 |
| 规模期 | 自建 Rust 边缘网关 + 云 CDN 协同 | 承担媒体转发、切片、AI 旁路、边缘代理 | 控制面、调度、商业化、数据治理 |

## 7. 关键取舍

- 不在 MVP 自研完整流媒体 CDN：成本和风险过高，云厂商能更快验证产品。
- 不全盘 Rust：业务迭代速度比极致性能更重要，Rust 聚焦确定会成为瓶颈的数据面。
- 不让投喂设备直接暴露 HTTP API：统一 MQTT over TLS + Broker ACL，降低设备公网暴露风险。
- 不把排行榜设计成单纯“吃得越多越好”：要引入健康上限、公益贡献、陪伴时长等更安全的指标。

### 8.2 物理部署演进

| 阶段 | 部署形态 | 关键说明 |
| --- | --- | --- |
| MVP | 单云、单 region、多 AZ；K8s 1 套；云厂商直播+CDN | 不做多活；DB 主从 + 异地备份；目标快速验证 |
| 成长期 | 单云双 region（华东/华北）+ 1 个边缘节点试点 | 控制面双活、数据面就近接入；Kafka 跨 region 异步同步 |
| 规模期 | 多云/多 region + 多边缘 PoP；按用户/设备亲和路由 | DB 分片 + 跨 region 灾备；CDN 多厂商；Rust 边缘大规模铺设 |

跨 region 的关键约束：

- 用户主资料、订单写入路由到“用户所属主 region”（按手机号归属或注册 region），避免双写冲突。
- 房间媒体由“房间所属 region”的边缘提供，客户端通过 DNS/调度服务就近接入。
- 设备 MQTT 接入按设备 ID 哈希到指定 Broker 集群，跨 region 漂移由设备控制面 + 重连解决，不依赖 Broker 自身跨 region 强一致。

### 8.3 接入与流量调度

| 流量 | 调度方式 | 说明 |
| --- | --- | --- |
| 业务 API | DNS 智能解析 + L4/L7 负载 + API 网关 | 按用户地理位置选最近 region；限流、熔断、灰度在网关 |
| 直播拉流 | DNS + CDN 调度 + 失败 fallback 协议 | 客户端拿到协议候选列表，按延迟/成功率自适应（见 02 第 4 节） |
| 实时长连 | Anycast/区域接入点 + 一致性哈希到具体网关实例 | 房间分片，同一房间的连接尽量落到同一组实例减少广播开销 |
| 设备 MQTT | 设备控制面下发 Broker 地址 + 失败回退列表 | 设备首次启动通过 HTTPS 获取 Broker 列表与证书 |
| 媒体推流 | 推流地址由房间服务签发，绑定 region | 防盗推 + 失败回源到备 region |

## 10. Rust + Go 协作时序（投喂为例）

```text
用户 (Web/App)
  | 1. POST /feed-requests
  v
API 网关 (Go)
  | 2. 鉴权 + 限流
  v
投喂服务 (Go)
  | 3. 校验 (Redis 原子冷却 + 健康规则)
  | 4. 落 feed_request 行 (MySQL)
  | 5. produce(feed.command.requested) -> Kafka
  v
设备控制消费者 (Go)
  | 6. 按 device_id 串行消费
  | 7. 生成 device_command + 签名
  | 8. publish MQTT /devices/{id}/commands
  v
边缘代理/投喂机 (设备/Rust 边缘代理)
  | 9. 执行 + 回执 MQTT /devices/{id}/acks
  v
设备状态服务 (Go)
  | 10. 更新 feed_request 状态
  | 11. produce(feed.command.acked) -> Kafka
  v
实时网关 (Rust)
  | 12. 房间广播 + WebSocket 推送
  v
通知服务 (Go)
  | 13. 推送/站内消息补齐 (App 后台/失联情况)
  v
媒体旁路 (Rust)
  | 14. 抓取出粮窗口截图 + 短视频
  v
对象存储/CDN
```

关键不变式：

- 用户请求路径上**没有任何 Rust 服务**；Rust 只在异步事件链与媒体面参与，保证业务可独立演进。
- 同一 `device_command_id` 经过 MQTT/Kafka 任何重投都必须最终只产生一次真实出粮（设备端 + 控制面双重幂等）。
- Rust 边缘代理只签发本地缓存命令、不持久化业务状态；离线恢复后由 Go 控制面对账。

## 11. 服务拆分边界

首期按“最小可独立部署”划分，避免一开始拆成几十个微服务：

| realtime-gateway | Go (MVP) → Rust (成长期) | WebSocket/QUIC 长连、弹幕广播 | Redis、Kafka |
| notification-service | Go | 站内消息、推送通道适配、频控 | MySQL、Redis、第三方 |
| payment-service | Go | 订单、回调、退款、权益 | MySQL、第三方 |
