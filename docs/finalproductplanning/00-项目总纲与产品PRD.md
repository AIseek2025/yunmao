# 00-项目总纲与产品 PRD

## 0. 文档定位

本文是 yunmao 项目的“最高纲领”：定义愿景、目标用户、核心价值、产品定位、PRD 主流程、状态机与异常路径、非目标和成功指标。所有其他文档（01–11）必须与本文不冲突；如有冲突，以本文为准并同步更新本文。

## 1. 项目愿景

yunmao 是面向中国大陆用户的“真实猫咪直播 + 远程投喂 + 社交陪伴 + 公益救助”平台。第一版同步交付 Web/H5 与 iOS/Android App 的核心体验：用户可以通过网页快速进入，也可以通过 App 获得更稳定的播放、推送提醒和长期陪伴关系；点击投喂后通过真实设备出粮，并在数秒内看到画面反馈。

核心不是“把 feed-the.cat 汉化”，而是把它验证过的真实互动感、低门槛观看、投喂冷却、房间/排行榜/聊天体验，升级为适合国内社交、支付、合规和高并发规模的云养宠平台。

## 2. feed-the.cat 可借鉴点

本次通过直接访问 `https://www.feed-the.cat/` 和公开检索可观察到：

- 站点包含 rooms、about、rankings、房间信息、聊天、房间投喂统计、用户资料、隐私政策等基础模块。
- 产品靠捐赠维持运行，强调“帮助猫咪和救助机构”，每个房间可配置是否允许聊天、是否允许投喂、投喂冷却时间、禁投喂时间窗、灯光触发策略。
- 站点明确使用冷却时间避免过度投喂，并在 FAQ 中解释碗空、碗满、猫不在镜头内等常见情况。
- 当前访问时直播区域提示维护或直播下线，因此直播画质、交互延迟和真实设备响应只能依据页面文案与公开报道推断。
- 公开报道提到：多房间、多摄像头、救助站接入、聊天、广播/电台、排行榜、未来猫咪检测和识别等方向。

对 yunmao 的启发：

- MVP 要把“真实反馈”做得比功能数量更可靠：点击投喂、设备确认、直播画面反馈、结果通知必须闭环。
- 冷却、禁投喂时间窗、房间级权限、设备状态必须是核心能力，而不是运营后台的附属项。
- 救助站/猫咖/猫主人接入流程必须产品化，不能长期依赖人工配置。
- 早期可以轻商业化或捐赠，但规模化必须引入支付、风控、成本核算和动物福利治理。

## 3. 目标用户

### 3.1 用户分层

| 用户 | 核心需求 | 产品承诺 |
| --- | --- | --- |
| 云吸猫用户 | 想看猫、互动、解压，但不能真实养猫 | 低门槛进入、高清直播、真实投喂反馈、可分享的温暖瞬间 |
| 已养猫用户/猫主人 | 展示猫、获得粉丝、管理设备、获取收益 | 标准化开播、设备状态、投喂上限、收益与数据看板 |
| 救助站/猫咖/高校猫协 | 获得曝光、猫粮支持、公益透明 | 专属机构页、公益记录、设备补贴、合规捐赠/赞助路径 |
| 宠物品牌/硬件厂商 | 精准触达宠物人群，验证智能设备 | 品牌房间、硬件接入标准、数据合作和广告位 |

### 3.2 市场背景（用于支撑商业假设，不替代正式市调）

| 指标 | 量级（公开口径/早期调研） | 对 yunmao 的意义 |
| --- | --- | --- |
| 中国城镇宠物猫数量 | 2024 年约 7000 万只量级，连续多年保持高速增长 | 供给侧（直播猫源）和需求侧（云吸猫用户）的天花板都足够大。 |
| 智能宠物用品渗透率 | 一二线城市约 30%+，自动喂食器/饮水机增速明显 | 设备供应链可借力，无需 0 到 1 自研。 |
| 单人户家庭 | 已突破 1.4 亿户，年轻独居人群是核心潜在用户 | 情感陪伴 + 低门槛云养精准击中独居青年。 |
| 海外参考 | feed-the.cat 上线后短时间内全球爆火、服务器多次过载 | 验证机制可行性，但需更稳健的高并发与合规底座。 |

以上数据来自早期调研（参见 `docs/earlyproductplanning/`），仅作为产品假设输入；商业化阶段应以正式市调和自有数据为准。

## 4. 核心价值

- 真实互动：用户的线上行为能驱动物理世界变化，形成比普通直播更强的参与感。
- 情感陪伴：通过固定猫咪、猫咪主页、关注提醒、投喂记录，建立长期关系。
- 公益可信：救助站房间将投喂、捐赠、物资消耗和直播画面关联展示。
- 高性能平台能力：面向未来数百万直播/投喂设备、数千万用户，底层从第一天按控制流、媒体流、数据流解耦设计。

## 5. 产品定位

第一阶段定位为“Web + App 同步上线的云养猫直播投喂平台”：

- 用户主链路：进入 Web 或 App 首页 → 选择猫咪房间 → 看高清直播 → 登录 → 点击投喂 → 设备出粮 → 看到反馈 → 生成记录/分享。
- App 增强链路：关注猫咪 → 接收开播/可投喂/投喂结果提醒 → 弱网下自动降级播放 → 回到房间继续互动。
- 运营链路：运营人员通过 Web 后台审核房间、设备、内容、支付和告警。
- 供给侧链路：猫主人/机构申请入驻 → 设备绑定 → 推流配置 → 房间审核 → 上线运营；首期可先由 Web 后台承接，App 端提供基础查看与通知。

## 6. MVP 范围

### P0 必须交付

- Web/H5 首页、房间页、猫咪主页、平台介绍页。
- iOS/Android App 首页、房间页、猫咪主页、登录、用户中心与基础通知入口。
- 手机号验证码登录，预留微信登录接口；同一账号可在 Web 与 App 登录并同步投喂记录、关注关系和权益。
- 单房间高清直播播放，支持至少 1080p/720p/480p 三档；Web 与 App 都必须具备协议降级和清晰度切换。
- 投喂按钮、全局冷却、用户冷却、房间禁投喂时间窗；Web 与 App 状态一致。
- 投喂设备 MQTT 指令下发、设备回执、失败重试、幂等处理。
- 基础弹幕/聊天室，含敏感词和频率限制。
- 投喂记录、房间投喂统计、Top feeders 榜单。
- 房间/设备管理后台的最小能力：开关投喂、设置冷却、查看设备在线；首期以 Web 后台为主。
- 通知服务：站内消息、Web 浏览器通知能力预留、App 推送通道设计与最小可用接入。
- 监控告警、日志追踪、基础内容安全审核。

### P1 上线后迭代

- 分享海报、邀请好友、每日任务、猫咪关注提醒。
- AI 猫咪入画检测、进食检测、精彩瞬间截图/短视频。
- 机构主页、公益房间、物资消耗展示。
- 微信支付/支付宝用于投喂券、会员或公益赞助。
- 多摄像头/多猫房间。
- App 能力增强：深度链接、后台播放/画中画评估、厂商推送优化、弱网播放策略细化、猫主人移动管理能力。

### P2 中长期

- 微信小程序、猫主人端增强 App、机构 SaaS 能力。
- 自研/认证硬件套件，开放设备 SDK。
- 大规模边缘媒体网关、Rust 流媒体节点、智能调度。
- 宠物品牌、电商、机构 SaaS、跨宠物品类扩展。

## 8. 非目标

- 第一版不做完整宠物社区信息流。
- 第一版不做复杂 App 社区、电商商城、猫主人完整工作台；App 首期聚焦观看、投喂、关注、通知和用户中心。
- 第一版不自研全链路 CDN；先用云厂商直播/CDN，保留未来 Rust 边缘节点替换或旁路增强能力。
- 第一版不鼓励无限投喂或“越多越好”的排行榜，所有增长玩法都必须受动物福利上限约束。
- 第一版不做激光笔、逗猫棒等会刺激猫咪的远程控制；不做 AI 猫语翻译等娱乐型 AI 主功能（参见 10 章商业化优先级）。
- 第一版不开放任意第三方设备直连云端；所有设备必须经认证流程接入（参见 04 第 2 节）。

## 9. PRD 核心流程

### 9.1 关键用户旅程（Web 与 App 共用）

```text
入口
  → 首页/大厅 (rooms)
  → 房间页 (room_id)
    ├─ 观看直播 (HLS/LL-HLS/HTTP-FLV/WebRTC)
    ├─ 弹幕/聊天
    ├─ 房间信息/设备状态
    └─ 投喂按钮
         ├─ 未登录 → 引导登录 (回跳保留 roomId 与意图)
         └─ 已登录 → 投喂请求 (异步状态机, 见 9.2)
  → 猫咪主页 (cat_id)
  → 用户中心 / 投喂记录 / 通知中心
  → 分享 (海报/链接/深链)
```

### 9.2 投喂请求状态机（PRD 视角）

```text
未登录 ──登录通过──┐
未登录 ──放弃登录──→ idle (按钮可见, 不可用)
                    │
已登录 ──点击投喂──→ submitting
   ├─ 服务端校验失败 → rejected(reason)
   │    ├─ 房间禁投喂窗 / 设备离线 / 冷却中 / 健康上限 / 风控拒绝
   │    └─ 文案分别可读: "稍后再喂" / "设备离线" / "冷却中,剩余 Ns" / ...
   ├─ 服务端校验通过 → accepted → queued
   │    └─ UI: "请求已接收"
   ├─ 设备指令已下发 → dispatched
   │    └─ UI: "正在出粮…"
   ├─ 设备执行成功 → succeeded
   │    └─ UI: "出粮成功" + 直播画面叠加 + 房间系统消息
   ├─ 设备执行失败 → failed(error_code)
   │    └─ UI: "出粮失败" + 补偿入口
   └─ 超时未回执 → uncertain
        └─ UI: "结果确认中" + 异步推送/站内消息补齐 (App 切后台仍需收到)
```

详细字段、幂等键、补偿路径参见 04 第 4 节。

### 9.3 异常路径与降级

| 场景 | 用户感知 | 系统行为 |
| --- | --- | --- |
| 直播离线 | 房间显示“直播维护中”，保留聊天与历史精彩瞬间 | 推流监控 → 通知猫主人；房间状态置为 `live_offline` |
| 设备离线 | 投喂按钮不可用，显示“设备离线” | 投喂服务直接 reject；关注用户按频控发送通知 |
| 弱网 | 自动降码率，必要时切协议 | 客户端 QoE 上报 → 服务端可主动下发更保守的播放配置 |
| 后台切换 (App) | 默认暂停视频，前台恢复后续播 | 进入后台时停拉媒体流；投喂状态由 ws/推送/拉取三路补齐 |
| 推送权限未开启 | 站内消息保底 | 不影响核心链路，引导但不强制开启 |
| 支付未上线 | 投喂按钮不显示付费投喂 | 订单 API 仍存在，灰度开关关闭 |
| 内容违规 | 房间下线/弹幕拦截 | 内容审核服务介入，记录审计日志 |
| AI 服务不可用 | 不显示“精彩瞬间”，但不影响投喂 | 旁路降级，主链路无依赖 |

### 9.4 可量化验收（PRD 级，工程级见 05）

| 流程 | 验收 |
| --- | --- |
| 首屏 | 进入房间页 ≤ 1.5s P95 看到首帧或明确状态 |
| 登录 | 手机号登录 ≤ 15s 完成，验证码 60s 冷却 + 风控 |
| 投喂 | 点击到“出粮成功” P95 ≤ 5s；端到端成功率 ≥ 99% |
| 弹幕 | 发送到广播 P95 ≤ 500ms；敏感词命中率经审核样本验证 |
| 通知 | 投喂结果 P95 30s 内到达任意端（站内/推送/Web 通道任一） |
| 一致性 | Web 与 App 上的冷却剩余时间、投喂记录、关注列表差异 ≤ 5s |

## 10. 未来端扩展性

第一版只交付 Web/H5 + iOS/Android App，但所有 API、事件与设计 Token 必须为以下端预留：

| 未来端 | 触达目标 | 一期需要保留的能力 |
| --- | --- | --- |
| 微信小程序 | 微信生态裂变、低门槛分享落地 | 同源账号体系（手机号/微信）、播放协议白名单、分享卡片字段、订阅消息通道 |
| 抖音/快手小程序 | 短视频生态投放后承接 | 与微信小程序共享数据接口，差异化 UI |
| TV/大屏 | 长时陪伴、客厅场景 | 高码率播放、遥控器适配、低交互模式（看猫为主） |
| 桌面端（Electron/Tauri） | 主播/机构端、运营后台增强 | 长连接稳定性、多窗口、本地通知 |
| Open API/SDK | 设备厂商、机构 SaaS | OAuth 授权、设备绑定 API、Webhook |

详细 API 与契约管理参见 08 第 3 节。
