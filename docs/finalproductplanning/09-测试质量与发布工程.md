# 09-测试质量与发布工程

本文定义 yunmao 项目的测试金字塔、关键质量门禁、QoE 与直播质量评测、压测、灰度与回滚、混沌工程、设备/IoT 测试方法。是 05（Phase 里程碑）与 08（多端工程）的执行细则。

## 1. 质量目标

- **可用性**：任何一次代码合并都不应让北极星 SLO（00/06 第 9 节）退化。
- **可重现**：所有质量结果必须在 CI 中可复现，本地必须能跑核心测试套。
- **可灰度**：每次发布有清晰的回滚路径与监控门禁。
- **不打无准备之仗**：每个 Phase 入口要有压测、混沌、QoE 报告作为放量依据。

## 2. 测试金字塔

```text
                ╱╲
               ╱E2E╲                 跨端/端到端: 少而精, 真实环境
              ╱──────╲
             ╱集成测试╲              跨服务/对外依赖: 中量
            ╱──────────╲
           ╱  单元测试  ╲             绝大多数, 快速
          ╱──────────────╲
         ╱静态/契约/格式  ╲           秒级反馈, 强约束
        ╱──────────────────╲
```

### 2.1 单元测试

- 覆盖率门槛见 05 第 9 节；核心域（投喂状态机、冷却、签名、状态切换、播放器选择）必须 ≥ 85%。
- 不允许在单测中调用真实数据库 / 第三方；使用接口 + mock。
- 业务规则用“表驱动测试 + 属性测试（PBT）”覆盖边界条件。

### 2.2 集成测试

- 服务级集成：服务 + 依赖（数据库/Redis/Kafka/MQTT）跑在 docker-compose 或 testcontainers 内。
- 关键集成场景：
  - 投喂请求 → 队列 → MQTT → 设备模拟器 → 回执 → 状态更新 → 通知
  - 房间状态变更 → 实时网关广播 → 多端订阅一致
  - 支付下单 → 第三方 mock 回调 → 权益发放 → 退款补偿
- CI 时长目标 ≤ 15 分钟（核心套）+ 30 分钟（夜间全量）。

### 2.3 契约测试

- 客户端 SDK 由 contracts 仓自动生成；契约本身有 lint 和兼容性检查。
- 服务端必须有“contract test”验证响应 schema、错误码、HTTP 状态码与 OpenAPI 一致。
- 消费者端（Web/App）保留少量 contract test，校验对错误码和必填字段的容忍度。

### 2.4 端到端测试（E2E）

- Web：Playwright，关键路径用例 ≤ 30 个，在 CI 内运行 ≤ 10 分钟。
- App：Patrol/Maestro，关键路径用例 ≤ 20 个，CI 跑模拟器；周期性在真机农场（如 Firebase Test Lab）跑兼容性矩阵。
- 数据：使用专用测试账号 + 沙箱设备 + 沙箱支付。
- 场景：登录、首页、进入房间、投喂闭环、关注、通知、登出、错误页。

### 2.5 视觉/无障碍

- Web：Storybook + Chromatic / Loki 做视觉回归（非阻塞，重大改动时启用）。
- 无障碍：axe-core 在 CI 中扫描关键页面；移动端用 Flutter 的 a11y 检查。

## 3. 测试数据治理

- 测试账号、测试设备、测试房间在 staging 长期保留。
- 生产数据进入测试环境必须脱敏（手机号/支付/IP），并通过审批。
- “金标数据集”：投喂状态机/直播 QoE 等关键测试都有可复用的数据集，纳入 contracts/test-data。

## 4. 直播 QoE 质量评测

### 4.1 客户端 QoE 指标（详见 02 第 12 节）

每次发布前必须在 staging 跑“直播会话回放测试”：

- 启动 N 路模拟客户端拉流，运行 ≥ 10 分钟。
- 采集首帧、卡顿、切档、失败码、平均码率。
- 与上一版本做对比，回归超过阈值（如首帧 +200ms / 卡顿率 +0.5%）阻断发版。

### 4.2 推流端测试

- 推流端用 GStreamer 推送已知样本，对接到 media-edge；监测端到端延迟与丢包恢复。
- 模拟弱网（tc/netem）测试上行抖动、丢包、断网恢复。

### 4.3 端到端延迟“投喂感知”

- 设备模拟器在收到出粮指令时记录“真实出粮时间”。
- 客户端记录“画面叠加投喂动效”时间。
- 自动算 P50/P95，超 SLA（参见 02 第 1 节）报警。

## 5. 压测

### 5.1 工具

- HTTP：k6 / Vegeta（Go 友好）。
- gRPC：ghz。
- WebSocket：自研 Go 压测客户端（goroutine 集群）或 Artillery。
- MQTT：emqtt_bench、自研 Rust 客户端。
- 媒体：客户端拉流压测用 ffmpeg/gstreamer 并发；推流压测用 OBS 集群或 ffmpeg 多进程。

### 5.2 关键场景

| 场景 | 目标 (Phase 1 / 2 / 3) |
| --- | --- |
| 投喂创建 | 50 / 5k / 5 万 QPS，错误率 < 1% |
| 弹幕发送 | 200 / 1 万 / 10 万 QPS，端到端 P95 < 1s |
| 弹幕广播 | 单房间 5k / 5 万 / 50 万 在线 |
| 进入房间（HTTP+长连+播放配置） | 100 / 5k / 5 万 QPS |
| 设备指令 | 20 / 1k / 1 万 设备并发 |
| 拉流 | 1k / 10 万 / 100 万 并发 |
| 支付下单 | 5 / 200 / 2k QPS |

每次压测产出报告（含瓶颈、CPU/内存/网络、依赖响应、错误分类），归档到 `qa/perf-reports/`。

### 5.3 压测纪律

- 压测必须在与生产同构（缩规模）的环境进行。
- 严禁直接对 prod 进行未授权压测；定期 game day 走专门审批。
- 关键依赖（数据库/Kafka/Redis/MQTT）设置上限保护，避免压测把共享组件击穿。

## 6. 灰度与发布

### 6.1 发布门禁（与 05 第 9 节联动）

- CI 必通过（含集成 + 契约 + E2E 关键路径）
- 发布前演练核对 Runbook
- 灰度阶段监控指标全部绿色才能进入下一阶段
- 任意阶段触发自动回滚条件：错误率 > 阈值、P95 延迟 > 阈值、业务核心指标（投喂成功率/播放成功率）下降 > 阈值

### 6.2 数据库变更

- 双写/影子表/在线 DDL（gh-ost/pt-osc 或 PG `CONCURRENTLY`）。
- 删字段/改语义至少经历“新增 → 双写 → 切读 → 停写 → 删除”五步，跨多次发布。

### 6.3 配置中心

- 冷却、健康上限、推送频控、支付价格、风控阈值、灰度白名单等必须配置化。
- 每次配置变更走 PR review；高危配置（投喂总开关、支付开关）需双人 + MFA 审批。

## 7. 混沌工程

从 Phase 2 起每月 1 次 Game Day，至少覆盖：

| 故障 | 触发方式 | 期望行为 |
| --- | --- | --- |
| 单 AZ 故障 | 强制 K8s 节点 cordon + drain | 服务自动迁移，SLO 不退化 |
| 数据库主库不可写 | 切只读 | 业务降级为只读，关键路径仍可观看 |
| Redis 抖动 | 注入延迟/丢包 | 投喂降级（拒绝部分请求），不会重复出粮 |
| Kafka 抖动 | 注入延迟 | 事件延后，UI 显示“结果确认中” |
| MQTT Broker 重启 | 进程重启 | 设备自动重连；指令在队列等待 |
| CDN 异常 | 路由切到失效 CDN | 客户端 fallback 切换；卡顿率短暂上升后恢复 |
| 上游支付不可用 | mock 第三方错误 | 投喂走免费路径或拒绝；订单不丢 |

## 8. 设备/IoT 测试

### 8.1 设备模拟器

- 提供 Python/Rust 实现的“虚拟投喂机”和“虚拟摄像头”。
- 支持注入：随机延迟、回执失败、卡粮、粮量不足、离线再上线、固件版本不匹配。
- 模拟器规模化：万级并发的 MQTT 客户端做压力测试。

### 8.2 真机回归

- 关键固件版本必须经过：
  - 24h 持续运行
  - 重启/断网/电源拔插
  - OTA 升级 + 回滚
  - 高低温稳定性（实验室抽样）

### 8.3 端到端真机演练

每周一次：选 1 个真实房间，运营人员触发一系列场景（卡粮、断网、夜间禁投），验证端到端体验。

## 9. 安全/合规测试

- 依赖漏扫：Trivy/Grype/Snyk，每次构建。
- 静态扫描：CodeQL、gosec、cargo-audit、Dart linter。
- 密钥扫描：gitleaks 阻断 PR；keychain/Vault 注入。
- 渗透测试：Phase 1 前一次基础测试；Phase 3 增加支付/隐私专项；每年一次第三方渗透。
- 合规自动检查：隐私 SDK 清单 diff、权限弹窗截图归档、上架前自查表。

## 10. 测试组织

- 每个服务/客户端开发负责单元测试与集成测试。
- QA 负责跨服务集成测试、E2E、压测脚本主干、自动化平台。
- SRE/Ops 负责混沌演练、容量预测、容量基线。
- 安全/合规由专人或兼职负责，定期复盘安全事件。

## 11. 报告与归档

每个 Phase 出口至少归档：

- 测试覆盖率与 E2E 报告
- 直播 QoE 报告
- 压测报告（含瓶颈与改造计划）
- 安全扫描结果
- 应急演练记录（事件 + 复盘）
- 灰度发布的关键指标曲线

所有报告进入 `yunmao-docs/reports/{phase}/`，并在 README 中索引。
