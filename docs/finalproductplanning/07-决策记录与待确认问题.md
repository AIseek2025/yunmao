# 07-决策记录与待确认问题

## 1. 已确定的关键决策

### ADR-001：Phase 1 同步交付 Web/H5 与 App 核心体验

决策：第一版同步交付 Web/H5 与 iOS/Android App 的核心体验；小程序作为后续增强阶段推进。

理由：

- Web/H5 链接传播门槛最低，适合首访、分享和搜索承接。
- App 更适合高频回访、关注提醒、投喂结果推送、弱网播放和长期陪伴关系。
- 账号、直播播放、投喂、支付、通知、设备状态和运营后台从第一天按多端共用设计，避免后续为 App 重做 API 和状态机。
- App 首期聚焦核心链路，不做复杂社区、电商和猫主人完整工作台，控制研发范围。

替代方案：

- 只做 Web：开发更快，但无法验证推送提醒、移动弱网、App 留存和长期陪伴体验。
- 先做小程序：微信裂变更强，但直播、支付、类目审核会拖慢早期验证。
- 先做原生双端 App：体验最可控，但首期人力和迭代成本过高。

### ADR-002：Go 做业务控制面，Rust 做性能数据面

决策：Go 负责用户、房间、投喂、支付、运营等业务服务；Rust 负责媒体边缘、实时网关、视频处理、AI 推理适配等高性能模块。

理由：

- Go 业务生态成熟、开发效率高。
- Rust 适合低延迟媒体包、零拷贝、高密度连接和边缘长期运行。
- 分层后可按瓶颈逐步引入 Rust，不拖慢 MVP。

替代方案：

- 全 Go：MVP 可行，但未来媒体和极端低延迟场景可能遇到 GC 和内存压力。
- 全 Rust：性能强，但业务迭代和团队成本高。

### ADR-003：MVP 使用云直播/CDN，不自研完整直播分发

决策：MVP 采用阿里云/腾讯云等成熟直播与 CDN，保留 Rust 媒体边缘演进路径。

理由：

- 云直播能快速获得转码、录制、CDN、内容安全能力。
- 自研 CDN 对早期产品验证没有必要，且成本和稳定性风险高。

替代方案：

- 自建 SRS 集群：可控性强，但需要更多运维与容量经验。
- 直接 WebRTC 全量分发：低延迟好，但成本和调度复杂，不适合所有房间。

### ADR-004：投喂必须异步队列化，设备按 device_id 串行执行

决策：用户投喂请求通过 Go 校验后入队，同一设备消费者串行下发 MQTT 指令。

理由：

- 防止高并发点击放大为设备重复出粮。
- 便于幂等、补偿、审计和限流。
- 可在高峰期做排队或聚合投喂。

替代方案：

- API 直接调用设备：体验看似更快，但不可抗洪峰，动物福利和安全风险高。

### ADR-005：动物福利优先于增长玩法

决策：排行榜、任务、分享和付费投喂都必须受冷却、每日上限和设备状态约束。

理由：

- 真实动物是产品核心，过度投喂会造成健康和舆论风险。
- 长期品牌信任比短期投喂次数更重要。

替代方案：

- 强化“大胃王”玩法：传播性强，但风险不可接受。

### ADR-006：成长期开始引入多 CDN，规模期自建 Rust 媒体边缘

决策：MVP 单 CDN；成长期引入双 CDN 主备 + 客户端 fallback；规模期叠加自建 Rust 媒体边缘 PoP（参见 02 第 13 节）。

理由：

- 多 CDN 大幅降低单厂商不可用风险，并通过竞价降低长期成本。
- 自建边缘在规模期能显著降低带宽成本、提升热门房间延迟与可控性。

替代方案：

- MVP 即多 CDN：调度与对账成本对早期没收益。
- 完全依赖单云：长期成本与稳定性风险不可接受。

### ADR-007：事件总线统一为 Kafka + CloudEvents 信封

决策：所有跨服务异步事件经 Kafka，使用 CloudEvents 1.0（或等价自定义信封）+ Protobuf 负载；Schema 在 schema-registry 集中治理。

理由：

- 多语言（Go/Rust/前端 QoE 上报）需要统一信封与版本化策略。
- 集中治理避免“漂移字段长期存活”引发不可逆复杂度。

替代方案：

- RocketMQ：国内生态成熟，可作为备选；若团队更熟悉 RocketMQ，可在 MVP 替换 Kafka，但信封/Schema 治理原则不变。
- 直接走 gRPC 同步：耦合过强，无法支撑高并发削峰与重放。

### ADR-008：微服务初始按“可独立部署的业务边界”划分，避免提前微服务化

决策：MVP 服务清单按 01 第 11 节切分（~13 个服务），不再细分；后续按瓶颈/团队规模再拆。

理由：

- 一开始拆几十个微服务会拖慢迭代和招聘曲线。
- 服务必须有清晰的数据所有权和契约，避免“分布式单体”。

替代方案：

- 单体起步：迭代最快但后续解耦成本高，且媒体/IoT 与业务必须早分离。
- 极致微服务：早期不必要，且对运维要求过高。

### ADR-009：支付分层 —— 订单/权益/补偿先于真实扣款

决策：Phase 1 完成订单、权益、补偿三层 API 与状态机；Phase 3 才上线真实扣款（投喂券/会员/赞助至少一种）。

理由：

- 投喂强依赖权益与补偿（出粮失败必须能退券/退金额），状态机必须从首期固定，避免商业化时重做投喂链路。
- iOS 虚拟权益需要 IAP，端差异化在 03/10 章约束。

替代方案：

- 上来就接真实支付：合规与对账压力过大，可能拖慢 MVP。
- 完全不规划支付：商业化阶段需重写投喂状态机，风险极高。

### ADR-010：App 首期推荐 Flutter，单团队覆盖 iOS + Android

决策：App 推荐 Flutter，单团队同步交付 iOS + Android；若 Phase 0 验证发现 Flutter 在播放/推送/弱网上存在不可接受问题，则降级为原生双端各 1 人。

理由：

- 单团队、单代码库，迭代速度与 Web 同步。
- 主流播放器、推送、深链、权限的 Flutter 插件生态可用。

替代方案：

- React Native：团队若已有强 RN 能力可选。
- Tauri Mobile：移动生态尚不成熟，首期不选。
- 原生双端：体验最可控但首期人力翻倍。

## 2. 本轮架构建议 / 临时决策

以下为 2026-05-24 底层核心模块复检后的架构负责人建议。状态为“临时决策”，可直接指导下一轮开发；其中投喂安全默认值仍需业务、运营或兽医最终确认。

### ADR-011：MVP 自研直播链路采用 RTMP 输入 + HTTP-FLV 输出

决策：MVP 自研链路仅承诺 RTMP 输入 + HTTP-FLV 输出，同时在 media-edge 预留 LL-HLS 切片器接口；WebRTC 暂缓。

理由：

- 当前 Rust 底座已形成 RTMP publisher、房间 pipe、HTTP-FLV 拉流的最小闭环。
- HTTP-FLV 足够支撑桌面 Web 和 Android 低延迟演示，LL-HLS 后续补齐 iOS/H5/CDN 兼容。
- WebRTC 会提前引入 SFU、NAT、移动端耗电和成本复杂度。

替代方案：完全走云直播 HLS/LL-HLS；或直接 WebRTC。若强互动房间明确要求 <1 秒、HTTP-FLV/LL-HLS 无法达标，再重新评估。

### ADR-012：事件总线推荐 Kafka

决策：跨服务异步事件默认采用 Kafka + CloudEvents 信封；RocketMQ 作为中国云厂商托管或团队强经验场景下的备选。

理由：

- Kafka 与 Schema Registry、ClickHouse、Flink、CDC、离线分析链路生态更成熟。
- 投喂、设备、QoE、计费、运营事件需要削峰、重放和多消费者订阅。
- 招聘、排障资料和本地开发验证更稳。

替代方案：RocketMQ、NATS JetStream、纯 gRPC 同步。若目标云厂商托管 RocketMQ 明显更优，或团队已有强 RocketMQ 运维能力，可重新评估。

### ADR-013：主数据库 PostgreSQL 起步，不直接上 CockroachDB

决策：MVP 主库推荐 PostgreSQL；通过 `region_id`、业务分片键、读写分离、多活预案为未来 CockroachDB/分布式 SQL 留迁移口。

理由：

- MVP 单 region、多 AZ 足够，PostgreSQL 能覆盖用户、房间、投喂、设备、订单核心数据。
- CockroachDB 会提前引入分布式事务延迟、SQL 兼容性和运维复杂度。
- 保留区域字段和分片键即可降低未来迁移成本。

替代方案：CockroachDB、MySQL、TiDB。若 Phase 2 出现明确多 region 强一致写入需求，再重新评估。

### ADR-014：引入 Buf CLI，Buf Schema Registry 暂缓

决策：下一阶段引入 Buf CLI，启用 lint、breaking check 和 codegen 验证；BSR 等多仓协作和 SDK 发布边界明确后再上。

理由：

- 当前 proto 脚本只校验文件和语法，不能防止破坏性字段变更。
- Buf CLI 能先建立本地和 CI 契约纪律，成本低。
- BSR 会引入账号、权限、远端发布流程，早期单仓阶段可暂缓。

替代方案：继续手写结构体、直接上 BSR、或 protoc 自维护脚本。若 proto 拆成 contracts 仓或开始发布多端 SDK，再重新评估 BSR。

### ADR-015：投喂安全默认值从保守值起步

决策：MVP 默认推荐 `room_cooldown=30s`、`per_user_room_cooldown=60s`、`per_cat_daily=12g`，并强制后台可配置、按房间/猫/设备覆盖。

理由：

- 动物福利与舆论风险优先于互动增长，默认值应偏保守。
- 当前代码与交付报告中的默认值不一致，需要收敛到单一配置源。
- 不同猫只和设备出粮精度差异大，硬编码默认值不能长期存在。

替代方案：维持更宽松默认值，或按活动临时放宽。最终默认值需业务/兽医确认；若真实试点证明过严，可灰度放宽。

### ADR-016：WebSocket 鉴权分离登录身份与房间订阅权限

决策：user-svc 负责签发登录 JWT；room-svc 负责签发短期 room subscription token；WebSocket 网关校验 token 后允许订阅。游客只能只读订阅公开视频与有限事件，投喂、弹幕、关注、支付等写操作必须登录。

理由：

- 登录身份和房间权限分离，避免长 JWT 承载复杂权限。
- room token 可短 TTL、按房间和协议控制，未来可复用到 CDN/播放鉴权。
- 游客只读保留首访转化，不放大内容安全和滥用风险。

替代方案：网关直接验 user JWT、完全禁止游客、完全匿名。若边缘/API Gateway 统一鉴权落地，可重新收敛 token 方案。

### ADR-017：立即添加最小 Web demo 作为回归工具

决策：下一轮建议在 `clients/web-demo` 放最小 Web demo，覆盖 flv.js/mpegts.js 拉流、WebSocket 订阅和投喂按钮；它只作为联调/回归工具，不作为正式前端。

理由：

- 当前底座缺少可重复打开的端到端观测面。
- curl + websocat 不适合产品、QA、运营自测。
- demo 能尽早暴露端口、CORS、播放兼容性和事件时序问题。

替代方案：暂不做客户端、直接启动正式前端工程、只用 Postman 集合。若正式 Web/H5 已覆盖同等 PoC 回归能力，可取消单独 demo。

### ADR-018：MQTT broker MVP 自建 EMQX 5（详见 dev/adr/0012）

决策：device-svc 与硬件设备之间走 MQTT v3.1.1；自建 EMQX 5 单节点起步，生产期演进到 EMQX 集群或云 IoT 托管。device-svc 充当 Kafka↔MQTT 协议网关。

理由 / 替代 / 影响：见 `docs/dev/adr/0012-mqtt-broker-and-bridge.md`。

### ADR-019：WebSocket 网关采用 Redis Pub/Sub 跨实例扇出（详见 dev/adr/0013）

决策：gateway 引入 `Fanout` trait，默认 `LocalFanout`；多实例上线时切 `RedisFanout`（`yunmao:fanout:{room_id}` + `yunmao:room:{room_id}:online` SET），Kafka 已分发的事件不再走 Redis。

理由 / 替代 / 实测：见 `docs/dev/adr/0013-ws-fanout.md` + `docs/dev/03-third-iteration-deliverable.md` §4.E。

### ADR-020：身份默认 RS256 + JWKS，HS256 兼容到第五轮（详见 dev/adr/0014）

决策：user-svc / room-svc 默认 RS256；私钥来源走 `KeyProvider` 抽象（HS256 / RSA PEM / RSA ephemeral / KMS-TODO）；gateway 5min TTL 拉 JWKS；HS256 保留作为 dev / 兜底，第五轮再下线。

理由 / 替代 / 安全注释：见 `docs/dev/adr/0014-identity-and-keys.md`。

## 3. 技术待确认问题

- Web 前端框架最终选 React 还是 Vue 3？建议按现有团队熟悉度决定，文档不强制。
- App 首期是否确认采用 Flutter？当前建议 Flutter；若团队已有强 React Native 或原生能力，需要在 Phase 0 输出取舍结论。
- App 首期推送通道覆盖哪些 Android 厂商？是否先采用云厂商聚合推送？
- MVP 云厂商选型：阿里云直播、腾讯云直播或自建 SRS + CDN？
- 第一批摄像头和投喂机型号是什么？是否支持 RTSP/RTMP、本地控制、余粮/卡粮检测？
- MQTT Broker 采用自建 EMQX 还是云 IoT 平台？MVP 临时决定自建 EMQX 5（见 ADR-018 / dev/adr/0012），上量后是否切云 IoT 仍需在 Phase 2 决策。
- AI MVP 是先用云视觉服务，还是自建轻量模型？
- PostgreSQL 临时决策是否升级为正式主库决策？若业务坚持 MySQL 或云数据库托管方案，需要在 Phase 0 输出取舍结论。
- 是否需要一开始接入微信登录，还是先手机号验证码？App 上架与微信开放平台配置也需同步评估。

## 4. 产品待确认问题

- 平台正式名称是否确定为“云养猫/yunmao”？是否需要英文名和域名策略？
- MVP 首批房间来自个人家庭、猫咖、救助站还是内部样板间？
- 是否第一版开放游客弹幕？feed-the.cat 支持按房间配置，yunmao 建议默认登录后发言。
- 基础投喂是否免费？如果免费，每个用户每日免费次数是多少？
- 是否第一版开放真实支付？建议 Phase 1 至少固定订单和权益 API，真实扣款可在 Phase 3 开启；若需要验证商业化，可先做小额投喂券或赞助。
- App 首期是否上架应用商店，还是 TestFlight/内测分发？若涉及虚拟权益售卖，需确认 Apple IAP 与微信/支付宝边界。
- 排行榜采用“投喂贡献榜”还是“健康陪伴榜”？建议避免“最贪吃”成为唯一公开核心榜。

## 5. 合规待确认问题

- 公司主体、域名、ICP备案计划是否已确定？
- 是否涉及公益募捐？如涉及，需确认合作慈善机构或改为“机构赞助/物资支持”。
- 是否允许未成年人注册和付费？建议默认限制未成年人消费。
- 家庭摄像头是否可能拍到人脸、住址、室内隐私？需要入驻审核和画面规范。
- 直播内容资质由云厂商能力覆盖到什么程度？平台是否需要额外申请资质？
- App 隐私合规材料、权限弹窗、SDK 清单和应用商店审核要求由谁负责维护？

## 6. 近期建议

1. 先确定 MVP 硬件套件和云直播方案，做 Phase 0 样机闭环。
2. 在 Phase 0 同时确定 App 技术栈、播放 SDK、推送方案和应用分发方式。
3. 并行设计 Go 服务的核心数据模型和 API，确保账号、投喂、直播、支付、通知、设备状态从首期服务 Web 与 App。
4. 提前建立动物福利规则和运营后台开关，避免后续补救。
5. 对 feed-the.cat 继续跟踪，但只借鉴机制，不复制其纯捐赠和弱合规模式。
6. 优先建立契约仓库（OpenAPI + Protobuf + 错误码 + 设计 Token），从第一天约束多端一致性。
7. 在 Phase 1 内完成关键 Runbook 草稿（投喂误触发、直播大面积故障、推送误发、数据泄露），并演练至少 1 次。

## 7. ADR 维护规范

- 每条 ADR 字段统一：编号、决策、理由、替代方案、状态（拟定/已采纳/已废弃）、影响范围、依赖文档。
- 新增 ADR 需在 PR 中关联代码改动或文档改动；已废弃 ADR 不删除，标记 `Superseded by ADR-NNN`。
- 文档作者 + 1 名跨团队评审通过即可采纳；架构级 ADR（媒体面、数据面、契约）需要架构组评审。

## 9. 第七轮待用户确认（建议本轮内收敛）
