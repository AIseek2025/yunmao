# 06-运维监控安全合规

## 1. SLO

| 能力 | SLO |
| --- | --- |
| Web/App/API 可用性 | 月可用性 ≥ 99.9% |
| 直播播放成功率 | ≥ 98% MVP，成长期 ≥ 99% |
| 投喂请求服务端接收成功率 | ≥ 99.5% |
| 设备真实出粮成功率 | ≥ 99%（排除设备离线和运营关闭） |
| App 推送可达与深链成功率 | MVP 建立基线，成长期 ≥ 95% |
| 支付订单状态一致性 | 100%，不允许丢单 |
| 关键告警响应 | P0 5 分钟内响应，P1 30 分钟内响应 |

## 2. 监控体系

### 业务监控

- 房间在线数、观看人数、投喂次数、投喂成功率。
- 登录转化、投喂转化、分享转化、留存。
- Web 与 App 的核心漏斗拆分：访问/安装、注册、首播、首喂、关注、回访。
- 订单成功率、支付回调延迟、退款/补偿数量。
- 房间投喂开关变更、冷却配置变更。
- 通知发送、到达、点击、退订、频控命中。

### 媒体监控

- 推流在线率、推流码率、推流重连、丢帧。
- 播放成功率、首帧时间、卡顿率、协议降级次数。
- App 前后台恢复、弱网降级、机型/系统版本播放失败率。
- CDN 带宽、回源率、边缘命中率、单房间成本。

### 设备监控

- MQTT 在线数、连接成功率、断连率。
- 指令 ack 延迟、执行成功率、超时率。
- 粮量、电量、卡粮、重启、固件版本。
- 摄像头在线但投喂机离线、投喂机在线但直播离线等不一致状态。

### 系统监控

- API P50/P95/P99 延迟、错误率、限流次数。
- Redis 命中率、热点 key、内存和连接数。
- Kafka/RocketMQ 消费积压、重试、死信。
- 数据库 QPS、慢查询、复制延迟、连接池。

## 3. 告警分级

| 等级 | 示例 | 处理 |
| --- | --- | --- |
| P0 | 大面积无法播放、投喂误触发、支付丢单、数据泄露 | 立即止血，必要时全站关闭投喂 |
| P1 | 多房间设备离线、消息队列积压、CDN 回源异常、App 大面积无法登录或推送误发 | 30 分钟内处理，发布状态说明 |
| P2 | 单房间直播下线、AI 处理积压、分享生成失败 | 工作时间处理 |
| P3 | 报表延迟、非核心活动异常 | 排入迭代 |

## 4. 应急预案

### 投喂异常

1. 全局关闭投喂入口或指定房间关闭投喂。
2. 停止设备指令消费者，保留请求入队或直接拒绝。
3. 检查最近指令和设备回执，确认是否重复出粮。
4. 对受影响用户执行投喂券/余额补偿。
5. 复盘冷却、幂等和设备状态机。

### 直播异常

1. 切换备用播放协议或备用 CDN。
2. 将异常房间标记为维护中，展示历史精彩瞬间。
3. 通知猫主人检查摄像头/网络。
4. 保留推流日志、播放器错误码和 CDN 数据用于复盘。

### App 通知异常

1. 暂停异常通知模板或指定通知类型。
2. 保留站内消息，不重复补发可能造成打扰的推送。
3. 核查 APNs、Android 厂商推送、token 失效和深链配置。
4. 对误发、漏发、频控失效进行用户影响评估和复盘。

### 支付异常

1. 暂停支付入口，保留订单查询。
2. 停止自动发放权益，等待回调校验或主动查单。
3. 对账后补发或退款。
4. 所有支付异常必须有财务流水和审计日志。

## 5. 安全要求

### 账号与权限

- 手机号、微信 union_id 等敏感标识加密或哈希存储。
- App push token、设备标识、系统版本等按个人信息处理，最小化采集和脱敏展示。
- 管理后台必须启用 MFA、RBAC 和操作审计。
- 运营高危操作二次确认：关闭房间、改冷却、退款、封禁、设备解绑。

### API 安全

- 全站 HTTPS，内部服务 mTLS 或服务网格。
- API 网关限流：IP、用户、设备、房间维度。
- 所有写接口需要幂等键或服务端去重。
- 防刷：验证码、行为频控、设备指纹、风险分。

### 设备安全

- 一机一证，证书可吊销。
- MQTT ACL 限定设备只能访问自己的主题。
- 指令 HMAC 签名、有效期、nonce 防重放。
- OTA 固件签名校验，禁止未签名固件。
- 设备出厂或入驻时写入唯一身份，不允许共享密钥。

### 媒体安全

- 推流 URL 带时效鉴权，防盗推。
- 播放 URL 按需鉴权，防盗链。
- 直播画面需支持人工关闭和内容举报。
- 对家庭房间默认隐藏精确地址，避免隐私泄露。

## 6. 合规要求

### 中国大陆基础合规

- 域名和网站主体完成 ICP 备案。
- 涉及经营性互联网信息服务、支付、直播、社区互动时，需要持续评估 ICP 证、网络文化经营、信息网络传播视听节目等资质边界；早期可优先接入持证云直播/内容安全服务。
- 用户协议、隐私政策、儿童/未成年人保护规则必须上线前完成。

### 个人信息保护

- 只收集完成服务所需的数据：手机号、昵称、头像、登录状态、投喂记录、支付订单。
- App 权限按需申请：通知、相册、麦克风、相机、定位不得在首启一次性索取；首期观看和投喂不应依赖定位、麦克风或相机权限。
- 明确数据用途，不将个人数据用于未声明广告或第三方共享。
- 提供账号注销、数据删除和隐私请求入口。
- 日志脱敏，手机号、支付单号、IP 等敏感字段按权限查看。

### 内容安全

- 弹幕、评论、昵称、头像需审核。
- 直播画面接入内容安全检测，涉及违法、虐待动物、隐私泄露等情况立即下线。
- 举报入口必须在房间页可见。

### 支付与公益

- 支付回调验签、金额服务端校验、订单不可篡改。
- 未成年人消费限制和大额消费二次确认。
- 公益捐赠不能由平台自行随意宣称，需与合规机构合作或明确“赞助/物资支持”性质。
- 猫主人/机构分账、提现、税务处理需合规留痕。

### 动物福利

- 每个房间必须配置每日投喂上限、冷却、禁投喂时间窗。
- 猫主人/机构可随时暂停投喂。
- 系统不以过度进食作为唯一增长目标。
- 设备故障、粮量异常、猫咪异常行为必须通知责任人。

## 7. 容量规划

MVP：

- 5-20 个房间。
- 1 万注册用户，1000-5000 同时在线。
- 单房间 500-1000 同时在线。
- Web/H5 与 App 同步承载核心流量，App 推送 token 规模按 1 万级设计。

成长期：

- 1000+ 房间。
- 百万注册用户，10 万级同时在线。
- 热门房间 1-5 万同时在线。

规模期：

- 百万级设备。
- 千万级注册用户，百万级同时在线。
- 直播带宽主要由 CDN 和区域边缘承载，业务 API 保持无状态水平扩展。

## 8. 发布与变更

- 所有服务必须支持灰度发布和快速回滚。
- 数据库迁移必须向前兼容，禁止上线时长时间锁表。
- 设备固件 OTA 分批发布，先测试房间再扩散。
- 冷却、投喂上限、支付价格、风控规则必须配置化并审计。

### 9.2 组件级 SLO（参见 01 第 9.3 节）

每个组件必须发布 SLO，并将“错误预算”可视化在告警面板。错误预算耗尽 → 暂停非必要变更，专注稳定性。

### 9.3 错误预算策略

- 错误预算用尽 → 暂停新功能上线 1 周；恢复需写入复盘记录。
- 错误预算消耗速率告警：日消耗 > 总预算 10% 触发 P1；> 30% 触发 P0。

## 10. 可观测性三柱细化

### 10.1 指标（Metrics）

- 采集：Prometheus + 服务自报；客户端 QoE 经网关聚合到 ClickHouse。
- 命名：`{domain}_{metric}_{unit}`，例如 `feed_request_success_total`、`media_stall_seconds_bucket`。
- 标签：必须含 `service`、`env`、`region`；业务标签使用低基数（房间维度可采样后再聚合）。
- 仪表盘分层：北极星仪表（5–10 个核心指标）→ 业务仪表（按 domain）→ 组件仪表 → 排障细节。

### 10.2 日志（Logs）

- 后端结构化 JSON（zap/slog），必含 `trace_id`、`span_id`、`request_id`、`user_id_hash`、`room_id`、`device_id`、`event`、`level`。
- 不能落库的敏感字段：手机号原文、支付完整卡号、设备私钥；必须哈希或脱敏。
- 客户端崩溃日志接入 Sentry/类等价服务，按 release/版本归集，PII 字段脱敏。

### 10.3 链路追踪（Tracing）

- OTel SDK 接入；采样率：MVP 5%，关键路径（投喂/支付）100%。
- 跨服务、跨语言（Go/Rust）传递 W3C TraceContext；MQTT/Kafka 在 header 中携带 trace 信息。
- 关键追踪点：投喂请求全链路、播放配置签发、设备指令分发、支付回调。

## 11. 应急 Runbook（精简版）

每个 P0/P1 事件必须有 Runbook（保存在 `ops/runbooks/`），结构如下：

```
1. 触发条件 / 告警来源
2. 第一响应人 (on-call)
3. 5 分钟止血动作
4. 30 分钟根因排查路径 (含日志/仪表盘链接)
5. 用户沟通模板 (Web 公告 / App 弹窗 / 客服)
6. 复盘模板
```

必备 Runbook：

- 全站直播大面积无法播放
- 投喂误触发（设备多出粮 / 同请求多次出粮）
- App 大面积无法登录或推送误发
- 支付掉单 / 重复扣款
- 数据泄露（敏感字段进入日志、被外部访问）
- 内容违规事件大面积出现
- 设备 MQTT Broker 不可用
- Kafka 大面积积压
- 数据库主从延迟告警 / 主库不可写
- CDN 单点故障 / 多 CDN 同时异常

## 12. 容量预算

每个 Phase 提交一份《容量预算表》，含：

- 各组件预估 QPS / 连接数 / 带宽 / 存储
- 单位成本（云厂商月单价）
- 单房间 / 单 DAU 单位成本估算（用于商业化定价输入，参见 10 章）
- 异常情景（×2、×5、×10 流量）的成本与降级方案

预算示例口径：

- CDN：按流量阶梯单价 × 月总流量 + 回源费用
- 直播转码：按并发分钟数（云直播）或自有转码机时
- 数据库：按实例规格 + 备份/容灾 + 备库
- Redis：按内存 GB 月费 + 高可用副本
- Kafka：按消息总量 + 存储天数
- 通知：APNs/厂商推送一般免费，但短信、营销推送计费
- 客服/审核人力：按内容量峰值估算

## 13. 客服与用户沟通

- 状态页（status.yunmao.live）公布历史与正在发生的事件。
- 客服工单接入：站内反馈 + 邮件；高峰期通过运营后台分级处理。
- 重大事件统一对外口径，避免不同端文案不一致；模板见 Runbook。

## 14. 日志/数据保留与合规

| 数据 | 保留期 | 备注 |
| --- | --- | --- |
| 应用日志（INFO+） | 30 天热 + 180 天冷 | 长期需求需审批 |
| 行为日志 | 12 个月 | 含登录、投喂、弹幕；脱敏 |
| 支付与订单 | 法定保留（≥ 5 年） | 财务/法务要求 |
| 设备遥测 | 90 天热 + 1 年冷 | 异常事件保留更久 |
| 直播录制 | 默认不保留；机构/审核需要时按事件保留 30 天 | 与隐私政策一致 |
| 弹幕原文 | 90 天 | 审核取证 |
| 投喂证书 | 永久 | 用户资产 |

跨境传输：默认数据不出境；如使用境外 SDK，必须在隐私政策中披露，并通过合规评估。
