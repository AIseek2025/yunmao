# 03-功能模块与多端一期规划

## 1. 功能模块拆分

| 模块 | 子功能 | 一期优先级 | 首期多端要求 |
| --- | --- | --- | --- |
| 首页/大厅 | 房间列表、热度、直播状态、猫咪标签、维护状态 | P0 | Web/H5 与 App 同步交付；App 增加关注入口和通知引导 |
| 房间页 | 直播播放、投喂按钮、冷却倒计时、弹幕、房间信息、设备状态 | P0 | Web 与 App 都必须完成观看、投喂、弹幕和状态同步 |
| 猫咪主页 | 档案、故事、照片、投喂记录、粉丝/关注、精彩瞬间 | P0/P1 | Web 适合分享落地页；App 强化关注和回访 |
| 投喂 | 资格校验、冷却、排队、设备回执、失败提示、投喂记录 | P0 | Web 与 App 使用同一投喂 API、状态机和补偿逻辑 |
| 账号 | 手机号验证码登录、用户资料、游客态、微信登录预留 | P0 | 同一账号体系支持 Web/H5/App；移动端使用 Bearer Token |
| 实时互动 | 弹幕、系统消息、投喂广播、敏感词、频率限制 | P0 | WebSocket/实时网关不绑定单端；App 后台恢复后补拉关键事件 |
| 通知 | 站内消息、开播提醒、可投喂提醒、投喂结果、设备异常 | P0/P1 | App 首期接入最小推送能力；Web 先支持站内消息和浏览器通知预留 |
| 排行榜 | 房间 Top feeders、日/周榜、健康约束提示 | P1 | 多端展示同源数据，避免刺激过度投喂 |
| 分享增长 | 投喂海报、邀请链接、首喂礼包、任务 | P1 | Web 链接承接传播；App 支持深链回房间 |
| AI 内容 | 猫咪入画检测、投喂截图、精彩瞬间 | P1 | 结果服务端生成，多端复用 |
| 猫主人后台 | 房间设置、设备绑定、冷却配置、数据看板 | P0/P1 | 首期以 Web 后台为主；App 端提供设备状态查看和告警接收 |
| 运营后台 | 房间审核、设备审核、内容审核、用户封禁 | P0/P1 | 首期以 Web 后台为主，API 与移动端用户产品隔离 |
| 支付商业化 | 投喂券、会员、公益赞助、订单 | P1/P2 | 订单、权益、补偿和退款从第一天按 Web + App 共用设计 |

## 2. 一期多端用户体验

### Web/H5

目标：降低进入门槛，适合搜索、社群链接、内容传播和运营后台承载。

一期内容：

- 顶部展示平台定位、当前在线房间数、今日投喂次数。
- 房间卡片展示猫咪封面、名字、城市/机构、直播状态、投喂状态、热度。
- 支持筛选：正在直播、可投喂、救助站、热门、新猫。
- 直播下线或维护时必须明确提示，不隐藏房间。
- 分享页、平台介绍、入驻申请和运营后台优先由 Web 承接。

### App

目标：承接高频用户和长期陪伴关系，提供比网页更稳定的回访、提醒和弱网体验。

一期内容：

- 首页/大厅、房间页、猫咪主页、登录、用户中心与我的投喂记录。
- 关注猫咪、开播提醒、可投喂提醒、投喂结果通知。
- 弱网自动降码率、前后台恢复播放、投喂状态持久化。
- 深链打开房间、分享链接回流 App 或 H5。
- 设备异常和投喂结果通过站内消息保底，推送权限未开启时不影响核心链路。

### 运营后台

目标：首期就能支撑真实房间运行，而不是依赖人工改库。

一期内容：

- 房间审核、设备绑定、开关投喂、冷却与禁投喂时间窗配置。
- 内容审核、用户封禁、弹幕记录查询。
- 设备在线、粮量不足、卡粮、直播离线告警。
- 支付、订单、补偿能力可以先做配置与状态预留，随商业化阶段开启。

## 3. 核心流程

### 投喂流程

```text
用户在 Web 或 App 点击投喂
  → 未登录则引导登录并保留原房间上下文
  → 客户端请求 POST /feed-requests
  → 服务端返回 accepted/queued/rejected
  → accepted/queued 时按钮进入等待态
  → 设备回执成功后播放动画和系统消息
  → 生成投喂记录，可查看截图或等待 AI 回放
  → App 用户离开前台时通过站内消息/推送补齐结果
```

前端文案必须区分：

- “请求已接收”：服务端通过校验，但设备还没执行。
- “出粮成功”：设备已回执。
- “出粮失败”：设备离线、超时、卡粮或被运营关闭。

### 直播播放流程

- Web/H5 根据浏览器能力选择 HLS、LL-HLS、HTTP-FLV 或 WebRTC。
- App 默认使用 HLS/LL-HLS 或云厂商低延迟播放器 SDK，重点房间再启用 WebRTC。
- 所有客户端通过同一播放配置接口获取协议列表、鉴权 token、清晰度和降级策略。
- 播放 QoE 必须按端、系统、网络、协议、房间维度上报。

### 通知流程

- 服务端统一产生通知事件：开播、可投喂、投喂成功/失败、设备异常、关注猫咪更新。
- Web 首期展示站内消息，浏览器通知作为能力预留。
- App 首期接入 APNs 和 Android 厂商推送中的最小可用链路；无法接入全部厂商通道时，必须有站内消息兜底。
- 所有通知必须支持退订、频控和深链回房间。

## 4. 中国化玩法优先级

### 一期可做

- 首次投喂证书：轻量电子证书，鼓励分享。
- 赛博功德值：救助站房间投喂获得公益积分，但不夸大公益效果。
- 中文弹幕表情：内置安全可控的常用猫咪梗。
- 投喂海报：猫咪截图、用户昵称、房间二维码/链接。
- App 关注提醒：提醒用户回来看常关注的猫咪，但必须控制频率。

### 暂缓

- 远程激光笔/逗猫棒：涉及动物刺激、设备安全和实时控制，等投喂链路稳定后再做。
- AI 猫语翻译：娱乐价值高，但不能影响主线开发。
- 猫咪 PK、大胃王强刺激玩法：需先建立健康投喂模型，避免舆论风险。
- App 完整社区、电商商城、猫主人复杂工作台：放到后续增强阶段。

## 5. 多端共享边界

- 登录会话使用标准 OAuth2/OIDC 思路，支持手机号、微信、小程序、App 多端统一账号。
- 所有业务 API 不绑定 Web cookie，移动端使用 Bearer Token，Web 可使用同源 Cookie 或 Token，但权限模型一致。
- 播放地址返回协议列表，由客户端选择最优协议。
- 投喂、支付、设备状态、通知、房间状态使用同一事件模型。
- 分享素材、投喂截图、精彩瞬间由服务端生成，Web、App、小程序复用。
- Web 与 App 不强行共享 UI 组件；共享设计 Token、接口契约、埋点字典和错误码。

| 端 | 页面/能力 | 说明 | 验收标准 |

## 7. 一期验收场景

- 游客打开 Web 首页或 App 首页，进入房间，能在 1.5 秒 P95 内看到首帧或明确的离线提示。
- 游客点击投喂，会被引导登录；登录后返回原房间继续投喂。
- 用户在 Web 与 App 任一端投喂成功后，设备实际出粮，房间内所有在线用户看到系统消息。
- App 在投喂等待期间切到后台再回来，仍能看到正确投喂结果；必要时由站内消息/推送补齐。
- 房间冷却中时，Web 与 App 看到一致倒计时。
- 设备离线时，投喂按钮不可用，运营后台有告警，App 关注用户可收到受控频率通知。
- 弹幕包含敏感词时被拒绝或替换，并记录审核日志。
- 运营人员可以关闭某房间投喂，Web 与 App 状态在 5 秒内同步。
- 支付能力即使首期不开启，也必须完成订单、权益和补偿 API 边界设计，避免后续重做投喂链路。

## 8. 设计约束

- 投喂是稀缺动作，不做无限点击。
- 所有“猫咪健康”“公益捐赠”相关文案必须可证明，避免营销过度。
- Web 首屏优先展示直播，不让活动弹窗干扰第一次观看。
- App 首期优先稳定观看、投喂和通知，不追求复杂社区功能。
- 分享不应强制，投喂完成后提供轻量入口即可。

## 9. 跨端共享与差异矩阵

| 能力 | Web/H5 | iOS App | Android App | 小程序（未来） | 说明 |
| --- | --- | --- | --- | --- | --- |
| 账号体系 | 同 | 同 | 同 | 同 | 手机号 / 微信 / 多端同 SSO；后端为 Bearer Token 主，Web 同源 Cookie 可选 |
| 房间列表/详情 | 同 | 同 | 同 | 同 | 同一 API；Web 用 SSR 提升首屏 |
| 播放协议 | HLS/LL-HLS/HTTP-FLV/WebRTC | HLS/LL-HLS (+云 SDK)、有限 WebRTC | 同 iOS | HLS/LL-HLS | 协议候选由服务端下发，客户端按能力选 |
| 投喂 | 同 | 同 | 同 | 同 | 同一 API；App 必须做后台/弱网态恢复 |
| 弹幕 | 同 | 同 | 同 | 同（订阅限频） | 长连接走 WSS；小程序受平台限制走轮询/WSS |
| 通知 | 站内 + 浏览器通知预留 | APNs + 站内 | 厂商推送 + 站内 | 订阅消息 + 站内 | 通知偏好同账号共享 |
| 分享 | 链接 + 海报 + Web Share API | 系统分享 + 海报 | 系统分享 + 海报 | 微信卡片 + 海报 | 海报由服务端统一生成 |
| 支付 | 微信 H5 / 支付宝 H5 / 扫码 | Apple IAP（虚拟权益）+ 微信/支付宝（实物） | 微信/支付宝 | 微信支付 | iOS 虚拟权益必须走 IAP，需做端差异（参见 10 第 5 节） |
| 后台播放 | 浏览器原生 | 默认暂停（首期） | 默认暂停（首期） | 不支持 | 后续评估画中画/通知栏播放 |
| 深链 | URL Scheme + Universal Link | Universal Link | App Link | 微信路径 | 一致的 `room_id` / `cat_id` 路径 |
| 离线/弱缓存 | 不强求 | 关注列表、最近投喂 | 同 | 不强求 | 首期最小可用，扩展见 Phase 4 |

跨端一致性目标：任意时刻同一账号在不同端看到的“关注列表、冷却剩余、投喂记录、通知偏好”差异 ≤ 5s。

## 10. 状态机一览

模块级状态机的契约由对应文档维护，但 UI 必须严格映射：

- **投喂请求**：见 04 第 4 节 + 00 第 9.2 节。
- **直播间状态**：`offline` / `connecting` / `live` / `maintenance` / `archived`，UI 需对应明确文案。
- **设备状态**：`unbound` / `bound` / `online` / `offline` / `error(reason)` / `disabled`。
- **房间投喂开关**：`open` / `paused(reason)` / `closed_by_owner` / `closed_by_admin` / `closed_by_window`。
- **聊天权限**：`open` / `login_required` / `paused` / `closed`，运营可按房间配置。
- **通知偏好**：每类型独立开关 + 全局静默时段；客户端不可绕过服务端配置。

任何 UI 显示的状态必须来自服务端推送/拉取，不能由客户端推断；客户端只做 fallback 文案。

## 11. 增长与玩法（一期/二期/暂缓）

### 11.1 一期可做（低风险、强支撑核心链路）

| 玩法 | 说明 | 动物福利约束 |
| --- | --- | --- |
| 首次投喂证书 | 投喂后生成电子证书 + 分享海报 | 不绑定连续日数压力 |
| 关注关系 | 关注猫咪后接收开播/可投喂提醒 | 频控限制 1 天 ≤ N 条通知 |
| 房间公益值 | 救助站房间显示“今日已被投喂 X 次” | 仅展示，不形成竞速 |
| 弹幕表情 | 内置安全可控的猫咪梗表情 | 不可付费解锁敏感词 |
| 简易任务 | 每日观看 1 个房间、关注 1 只猫 | 投喂任务必须遵守冷却 |

### 11.2 二期增强

| 玩法 | 说明 |
| --- | --- |
| 排行榜 | Top feeders 限“贡献时长 + 健康投喂次数”，不是“吃得最多” |
| 投喂海报模板 | 节气/品牌联动模板 |
| 邀请有礼 | 双向奖励（积分/免费投喂券），反作弊 |
| 拼团 | 投喂券 / 会员拼团 |
| 集体投喂目标 | 房间日目标进度条，达成全员特效（不诱导刷量） |
| 云认养 | 付费认养，部分权益（专属称号、提醒、海报模板） |

### 11.3 暂缓 / 不做

- 远程激光笔/逗猫棒（动物福利与安全）
- 猫咪 PK / 大胃王（违反健康约束）
- AI 猫语翻译（娱乐型，不在主线）
- 复杂社区/电商商城（资源不足）
- “虚拟工会”等高竞争 PvP 机制（容易导致刷投喂）

详细商业化与运营落地见 10 章。

## 12. 可访问性与本地化

| 维度 | 要求 |
| --- | --- |
| 文案 | 简体中文为主，预留繁体/英文资源文件结构 |
| 字号 | 默认遵循系统字号；提供大字号模式 |
| 对比度 | 关键按钮 ≥ AA 级对比度；冷却倒计时不能仅靠颜色区分 |
| 视障 | 投喂按钮、播放器、登录提供语义化标签；图标必须有 `aria-label` |
| 听障 | 弹幕、系统消息可关闭声音；视频不依赖音频传达投喂结果 |
| 老人机型 | 默认 720p，长按操作；不强制手势 |
| 网络环境差地区 | 默认 480p / 音频模式可选 |

## 13. 通用 UI 设计 Token

| Token | MVP 取值（建议） | 用途 |
| --- | --- | --- |
| `color.brand.primary` | 暖橙系 | 主按钮、强调元素 |
| `color.brand.accent` | 奶油黄 | 投喂、奖励、温暖场景 |
| `color.status.success` | 柔和绿 | 出粮成功、设备在线 |
| `color.status.warning` | 蜜桃橙 | 冷却中、维护中 |
| `color.status.danger` | 暖红 | 失败、违规 |
| `radius.md` | 12px | 默认卡片圆角 |
| `shadow.card` | 柔光浅阴影 | 卡片层级 |
| `motion.feedback` | 200–300ms 缓入 | 按钮反馈 |
| `motion.celebration` | ≤ 800ms | 出粮成功动效，不影响连点 |

具体取值由设计稿确定，Web 与 App 共享 Token 名称与语义（参见 08 第 5 节）。
