# 05-开发 Phase 里程碑与验收

## 1. 阶段总览

| Phase | 周期 | 目标 | 退出条件 |
| --- | --- | --- | --- |
| Phase 0 | 1-2 周 | 技术预研与样机闭环 | 一台设备完成直播 + 投喂 + 回执演示，并验证 Web/App 播放技术路径 |
| Phase 1 | 6-8 周 | Web + App MVP 内测 | 3-5 个房间稳定运行，Web 与 App 核心链路可观测 |
| Phase 2 | 6-8 周 | 公测与增长能力 | 分享、排行榜、基础运营后台、App 通知闭环上线 |
| Phase 3 | 8-12 周 | 商业化与机构接入 | 支付、机构房间、设备批量接入、多端权益一致性上线 |
| Phase 4 | 3-6 个月 | 多端增强与规模化 | 小程序、App 深度能力、Rust 数据面逐步接入 |

## 2. Phase 0：技术预研与样机

目标：证明真实世界闭环可控。

交付物：

- 一台摄像头 + 投喂机 + 边缘盒子的端到端 Demo。
- MQTT 主题、设备证书、设备回执原型。
- 一个最简 Web 房间页：播放直播、点击投喂、展示状态。
- 一个 App 技术样机：播放直播、登录态存储、投喂状态展示、推送/深链技术验证。
- 直播协议选型报告：HLS/LL-HLS/HTTP-FLV/WebRTC 在当前设备与网络下的延迟数据。
- Flutter 与原生/React Native/Tauri mobile 的首期取舍记录，确认 App 技术栈。

验收标准：

- 连续运行 24 小时无进程崩溃。
- 投喂指令成功率 ≥ 98%。
- 设备离线、卡粮模拟能被平台识别并关闭投喂。
- 直播首帧和卡顿数据能被采集。
- App 样机在 iOS/Android 至少各一台设备上完成播放、前后台恢复和投喂状态演示。

## 3. Phase 1：Web + App MVP 内测

目标：让少量真实用户在 Web 与 App 上都能完成“看猫、登录、投喂、互动、记录、接收结果”的完整体验。

交付物：

- Web/H5 首页、房间页、猫咪主页、登录页、用户中心。
- iOS/Android App 首页、房间页、猫咪主页、登录、用户中心、通知中心。
- Go 用户服务、房间服务、投喂服务、设备服务。
- MQTT Broker、Redis 冷却、消息队列、基础数据库。
- 房间管理后台：开关投喂、设置冷却、查看设备状态。
- 弹幕/系统消息。
- 多端通知服务：站内消息、App 推送 token、投喂结果通知、通知偏好。
- Prometheus/Grafana 基础看板。

验收标准：

- 3-5 个真实房间连续 7 天运行。
- Web 与 App 播放成功率均 ≥ 98%，首帧 P95 ≤ 1.5 秒。
- 投喂端到端成功率 ≥ 99%，P95 设备回执 ≤ 2 秒。
- 弹幕敏感词、频率限制生效。
- 所有投喂都有 `feed_request_id`、`device_command_id` 和设备回执记录。
- 同一账号在 Web 与 App 的关注、投喂记录、冷却状态一致。
- App 弱网降级、前后台恢复、投喂结果通知和深链回房间通过验收。

## 4. Phase 2：公测与增长

目标：验证用户增长和留存，不让增长冲垮设备与直播。

交付物：

- 投喂海报、邀请链接、首喂礼包。
- 房间 Top feeders、日/周榜。
- App 关注提醒、开播提醒、可投喂提醒和通知频控。
- AI 投喂截图和猫咪入画检测原型。
- 运营后台：房间审核、用户封禁、内容审核、告警处理。
- 压测报告：API、弹幕、投喂请求、播放入口。

验收标准：

- 支持 1 万同时在线观看，核心 API P95 ≤ 200ms。
- 单房间 5000 在线弹幕可用，必要时采样降级。
- 1000 人同时点击投喂时，真实设备不会重复出粮，前端状态一致。
- 分享链路能归因新用户来源。
- App 推送点击能回到对应房间，通知退订和频控生效。
- 客服/运营能在后台处理房间投喂开关和用户举报。

## 5. Phase 3：商业化与机构接入

目标：让平台具备可持续运营成本回收能力，并接入救助站/猫咖/品牌房间。

交付物：

- 微信支付/支付宝：投喂券、会员或公益赞助中的至少一种。
- 订单、退款、对账、补偿流程。
- Web 与 App 权益一致：投喂券、会员、订单和补偿状态同源。
- 机构入驻流程、机构主页、公益/赞助记录。
- 批量设备绑定和设备健康看板。
- 基础风控：支付风控、刷邀请、刷投喂、异常设备行为。

验收标准：

- 支付成功率、回调验签、订单状态机通过测试。
- Web 与 App 查询到的订单、权益和补偿状态一致。
- 设备批量接入 50+ 台后，离线/卡粮/粮量不足告警准确。
- 机构房间能展示资金/物资用途说明，不夸大公益承诺。
- 财务可导出订单、分账、退款和对账报表。

## 6. Phase 4：多端增强与规模化

目标：在 Phase 1 已有 Web 与 App 核心链路的基础上，增强小程序、App 深度能力和 Rust 数据面。

交付物：

- 微信小程序核心链路：看直播、登录、投喂、分享。
- App 增强能力：更完善的播放 SDK、厂商推送、画中画/后台策略评估、猫主人设备管理、离线草稿与本地缓存。
- Rust 实时网关或媒体边缘网关试点。
- 多区域部署和容量规划。
- AI 精彩瞬间、猫咪检测、投喂回放服务。

验收标准：

- Web、小程序、App 使用同一套账号、房间、投喂、支付、通知和设备 API。
- Rust 试点服务有明确收益：延迟、成本、连接数或稳定性至少一项明显改善。
- 支持 1000+ 房间、百万级注册用户的容量演练。
- 故障演练覆盖直播、投喂、支付、设备、数据库和消息队列。

## 7. 每阶段必须产出的工程材料

- 架构设计更新。
- API 文档和事件契约。
- 数据库迁移脚本。
- 压测报告。
- 监控看板和告警规则。
- 安全/合规检查清单。
- 灰度发布和回滚方案。

## 8. 风险与缓解

| 风险 | 阶段 | 缓解 |
| --- | --- | --- |
| 直播成本失控 | Phase 1+ | 多码率、默认 720p、热门房间成本看板、CDN 预算告警 |
| 投喂过量引发舆论 | 全阶段 | 冷却、每日上限、称重/余粮检测、运营暂停、公开动物福利规则 |
| 设备不稳定 | Phase 0+ | 认证硬件、回执状态机、离线告警、本地日志 |
| 支付合规复杂 | Phase 3 | 先做低风险投喂券/会员，公益捐赠与合规机构合作 |
| AI 误判 | Phase 2+ | AI 只做辅助，不作为唯一投喂依据；关键异常人工复核 |
| 旧文档冲突 | 全阶段 | 编号文档为主入口，变更必须同步更新 |
| 多端 API 漂移 | Phase 1+ | 契约 + 自动化测试 + 灰度门禁（参见 08/09） |
| 单云依赖 | Phase 2+ | 多 CDN + 关键服务可迁移评估（参见 07 ADR-006） |
| 运营人手不足 | 全阶段 | 自动化告警 + 后台审核流程化 + 关键能力开关化 |

## 9. 每阶段质量门禁（必须通过才能进入下一阶段）

| Gate | Phase 0→1 | Phase 1→2 | Phase 2→3 | Phase 3→4 |
| --- | --- | --- | --- | --- |
| 单元测试覆盖率（关键域） | ≥ 60% | ≥ 70% | ≥ 75% | ≥ 80% |
| 集成测试（投喂/直播/通知） | 主路径通过 | 包含异常路径 | 跨服务故障注入 | 全链路混沌 |
| E2E（Web + iOS + Android） | 烟囱用例 | 完整核心链路 | 加入支付与机构 | 加入小程序 |
| 压测（投喂 QPS / 在线观看） | ≥ 50 / 5k | ≥ 5k / 10–20 万 | ≥ 1 万 / 50 万 | ≥ 5 万 / 100 万 |
| QoE 报告 | 单房间 | 多房间多 region | 多 CDN 切换演练 | 跨 region 演练 |
| 安全 | 基础渗透 + 设备签名 | 全量 OWASP Top 10 | 支付/隐私专项 | 红蓝对抗 |
| 合规 | 隐私政策上线 | 通知/隐私 SDK 清单 | 支付/未成年人合规 | 多端合规 |
| 监控/告警 | 基础四金 | 业务/媒体/设备完整 | 多 region 监控 | 多云联邦 |
| 应急演练 | 关闭投喂演练 | 直播切流 + 投喂回滚 | 支付掉单恢复 | 跨 region 切换 |

详细测试与压测方法见 09 章。

## 10. 灰度与回滚

每个发布必须遵循：

1. **预发验证**：与生产同构的预发环境过 P0/P1 用例。
2. **金丝雀**：1% 服务实例或 1% 用户灰度，关键业务指标（投喂成功率、播放成功率、错误率、延迟）超阈值自动报警。
3. **按用户分层**：内部员工 → 内测白名单 → 1% → 10% → 50% → 100%；任意一层异常立即停推。
4. **按 region 分层**：先小 region/小机房滚动，再放量。
5. **按房间分层**：媒体/投喂相关变更，先空闲房间再热门房间。
6. **回滚预案**：每次发布必须附带回滚命令/脚本与数据回滚预案；数据库迁移必须可逆（或可前向兼容回退）。

不能灰度的变更（如数据库结构变更、密钥轮转）必须有专项窗口与 Runbook（见 06 第 4 节）。

## 11. 研发组织建议

按“小而强 + 端到端责任”划分团队：

| 团队 | 人力建议（MVP） | 主要范围 |
| --- | --- | --- |
| 平台/架构 | 1–2 | 网关、契约、可观测性、CI/CD、安全基线 |
| 业务后端（Go） | 2–3 | 账号/房间/投喂/通知/订单/运营 |
| 媒体/数据面（Rust） | 1–2 | 媒体接入、边缘试点、视频处理 |
| 设备/IoT | 1 | MQTT、设备控制面、固件/边缘代理 |
| Web 前端 | 1–2 | Web/H5、运营后台 |
| 移动 App | 2 | iOS + Android（Flutter 单团队，或原生各 1） |
| 设计 | 1 | UI/UX + 设计 Token |
| 产品/运营 | 1–2 | 产品、内容审核、机构对接 |
| 测试/QA | 1 | 自动化、QoE、压测 |
| 安全/合规 | 半人 / 兼职 | 隐私、应用商店、内容审核 |

总人力 12–18 人即可启动 Phase 1。规模期再按瓶颈拆解（媒体、风控、增长各独立小组）。

## 12. 每阶段必须的演练

| 演练 | 频率 | 说明 |
| --- | --- | --- |
| 全站关闭投喂 | Phase 1 起每月 1 次 | 验证“一键熔断”可在 1 分钟内生效 |
| CDN/直播切流 | Phase 2 起每月 1 次 | 主备 CDN 切换、协议降级 |
| 数据库主从切换 | Phase 2 起每季 1 次 | RTO/RPO 达标 |
| MQTT Broker 故障 | Phase 2 起每季 1 次 | 设备指令降级 / 拒绝 |
| 支付回调失败 | Phase 3 起每季 1 次 | 对账与补发 |
| 跨 region 切流 | Phase 4 起每季 1 次 | 用户/房间路由策略 |
| 数据泄露应急 | Phase 1 起每半年 1 次 | 通报、止血、复盘 |
