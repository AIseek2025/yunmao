# 08-多端工程组织与协作

本文件定义 yunmao 在 Web/H5、iOS App、Android App 同步交付（并为未来小程序、TV、桌面端、Open SDK 预留扩展）的工程组织、共享边界、契约管理、仓库结构、分支与发版、CI/CD。是 01（架构）与 03（功能模块）的工程落地补充。

## 1. 目标与原则

- **同源契约，端内自由**：API、事件、错误码、设计 Token、埋点字典在“契约仓库”集中管理；端内 UI 实现各自最优。
- **多端同步交付**：Phase 1 起 Web 与 iOS/Android App 必须保持核心链路同节奏发版；不允许某端长期滞后。
- **可灰度可回滚**：每个端、每个服务都能独立灰度/回滚；多端共享的契约改动有专门的兼容窗口。
- **小步快跑**：少而稳定的服务边界、可自动化的发版流水线，避免一上来就引入复杂多仓配置。

## 2. 仓库结构（建议）

采用“多仓 + 契约单仓”模式：

```text
yunmao-platform/                # 平台总仓 (issue/decisions/roadmap)
yunmao-contracts/               # 契约单仓: OpenAPI / Protobuf / 错误码 / 事件 Schema / 设计 Token / 埋点字典
yunmao-services-go/             # Go 微服务仓 (monorepo): user, room, feeding, device, payment, notification, admin, moderation, analytics ...
yunmao-services-rust/           # Rust 数据面仓: media-edge, media-processor, realtime-gateway (晚期), edge-agent
yunmao-web/                     # Web/H5 (含管理后台子项目)
yunmao-app/                     # 移动 App (Flutter, 单代码库支持 iOS/Android)
yunmao-device-firmware/         # 设备固件 (ESP32/边缘盒子)
yunmao-infra/                   # K8s/Terraform/Helm/Ops 脚本
yunmao-docs/                    # 本规划文档目录与设计文档
```

可选简化方案（小团队 < 10 人）：把 Go 与 Web 合并为一个 monorepo（含 `services/`、`web/`、`contracts/` 子目录），减少跨仓 PR 协同成本；规模化后再拆。

## 3. 契约仓库（yunmao-contracts）

唯一真理来源（Source of Truth）：

```text
contracts/
├── openapi/                    # 客户端 REST API 规范
│   ├── v1/
│   │   ├── auth.yaml
│   │   ├── rooms.yaml
│   │   ├── feeding.yaml
│   │   ├── notifications.yaml
│   │   ├── orders.yaml
│   │   └── admin.yaml
│   └── codegen/                # 生成 Web/Flutter/Go SDK
├── proto/                      # 内部 gRPC + Kafka 事件
│   ├── feeding/v1/feeding.proto
│   ├── device/v1/device.proto
│   ├── media/v1/media.proto
│   └── events/v1/feed_events.proto
├── errors/                     # 错误码 + i18n 文案
│   ├── codes.yaml
│   └── messages/{zh-CN, en-US, ...}.yaml
├── design-tokens/              # 颜色/字号/圆角/动效 (JSON/JS/Dart 多端导出)
├── analytics/                  # 埋点字典: 事件名 / 字段 / 取值
└── schemas/                    # CloudEvents 信封 + JSON Schema (前端 QoE 等)
```

CI 钩子：

- PR 合并前必须通过 `buf lint` / `spectral lint`（OpenAPI）/ `protolint`。
- 破坏性变更（删字段、变枚举语义）必须升 major 或新版本；自动化检测（buf breaking 等）。
- 通过 codegen 自动生成各端 SDK，并发布到内部包仓库（npm 私库、pub 私库、Go module proxy、Maven private）。

## 4. 客户端 SDK 分层

为了让 Web 与 App 始终“调用同一组业务能力”，提供分层 SDK：

| 层 | Web (TS) | Flutter (Dart) | 内容 |
| --- | --- | --- | --- |
| Transport | fetch/axios + WSS | http + web_socket_channel | URL、超时、重试、签名、X-Trace-Id 注入 |
| API | 由 OpenAPI 自动生成 | 同 | 类型化方法签名 + 错误码映射 |
| Domain | 手写 hook / repository | 手写 repository | 投喂状态机、播放协议选择、通知偏好等业务封装 |
| UI | React 组件 | Flutter Widget | 由各端自行实现，不共享 |

设计 Token 通过 Style Dictionary 生成：

- Web：CSS variables + TS 常量
- Flutter：Dart 常量
- 运营后台：Ant Design / 内部组件库主题包

## 5. 端内技术栈细化

### 5.1 Web/H5

- 框架：TypeScript + React 18（或 Vue 3，团队决定后写入 ADR）+ Vite。
- 状态：TanStack Query（数据）+ Zustand/Jotai（局部 UI 状态）。
- 路由：React Router / Vue Router；H5 路由保留 `/rooms/:id` 等关键 URL 用于分享。
- 播放器：抽象 `PlayerAdapter`，实现 hls.js / mpegts.js / WebRTC（云厂商 SDK）。
- 通知：浏览器通知（预留），Service Worker 视情况启用（避免 PWA 过度复杂）。
- SSR/SEO：房间分享页 + 平台介绍页用 SSR/SSG（Next.js/Nuxt），提升首屏与搜索。

### 5.2 移动 App（推荐 Flutter）

- 单代码库支持 iOS/Android（ADR-010）。
- 播放器：在 Flutter 端封装“播放协议适配器”，按平台调用：
  - iOS：AVPlayer（HLS/LL-HLS），WebRTC 经云厂商 SDK
  - Android：ExoPlayer（HLS/LL-HLS/HTTP-FLV），WebRTC 经云厂商 SDK
- 推送：firebase_messaging（GMS 地区）+ 国内厂商通道（小米/华为/OPPO/vivo）通过统一聚合（如个推/极光/友盟）或自建 push-router；iOS 走 APNs。
- 深链：Universal Link（iOS）+ App Link（Android）+ Custom Scheme fallback。
- 后台策略：默认进入后台 30s 后释放解码器；保留长连接接收事件。

### 5.3 运营后台（Web）

- 框架：React + Ant Design（或 Naive UI）。
- 共享 contracts 中的 `admin.yaml`、错误码、设计 Token。
- 鉴权：与用户体系完全隔离，独立 SSO + MFA。

### 5.4 Go 服务

- 框架：Gin/Fiber/CloudWeGo Kitex 任选；服务模板统一从 `yunmao-services-go/template/` 派生。
- 依赖注入与配置：viper + 配置中心；密钥从 Vault/KMS 拉取。
- ORM：sqlc/ent（避免运行时反射型 ORM 性能波动）。
- 测试：testify + dockertest（数据库/Kafka/Redis 容器化集成）。

### 5.5 Rust 服务

- 异步运行时：Tokio；服务框架：Axum/Tonic。
- 媒体管线：GStreamer 或 FFmpeg bindings + 自研 RTMP/SRT/HLS 模块。
- 测试：cargo test + criterion（基准）+ 模拟设备/客户端工具。
- 构建：cross/zigbuild 出多平台 binary；K8s 镜像基于 distroless。

### 5.6 设备/边缘代理

- 投喂机：ESP32/STM32 + 自研固件，统一 MQTT 客户端 SDK。
- 边缘盒子：ARM64 Linux + Rust 边缘代理；按需附加 GStreamer/AI 推理。
- OTA：差分包 + 签名校验，按房间灰度。

## 6. 分支与版本

### 6.1 分支模型（Trunk-based + 短分支）

- `main`：始终可发布；保护分支，必须 CI 通过 + 至少 1 名审阅。
- `feature/{ticket}-{slug}`：短生命周期（≤ 5 天），合并前 rebase。
- `release/v{X.Y}`：发版前从 `main` 拉出，仅接受 hotfix 与必要打包变更；发完合回 `main`。
- `hotfix/v{X.Y.Z}`：从 `release/v{X.Y}` 拉出，修完合回 `release` 与 `main`。

不使用长生命周期 develop 分支，避免合并地狱。

### 6.2 版本号

- 客户端：SemVer `MAJOR.MINOR.PATCH`；移动端再加 `build` 号（CI 自增）。
- 服务端：与时间窗对齐 `YY.MM.X`；契约与服务版本独立演进。
- 契约：每个文件首行声明 `version: vN`；OpenAPI/Proto 文件路径含版本目录。

### 6.3 兼容窗口

- 服务端 API：旧版本至少保留 6 个月或 2 个移动端大版本（取较长）。
- 移动端最低支持系统：iOS ≥ N-2，Android ≥ API 23（按数据每年评估）。
- 强制升级仅在严重安全/合规场景使用，必须有降级 fallback（如只读模式）。

## 7. CI/CD

### 7.1 CI 流水线（按仓库）

| 仓库 | 必跑步骤 |
| --- | --- |
| contracts | lint（buf / spectral / protolint）+ breaking change check + codegen 验证 + 发布 SDK（合并到 main 时） |
| services-go | `go vet` + `golangci-lint` + 单元 + 集成（dockertest）+ 镜像构建 + SBOM + 漏扫 |
| services-rust | `cargo fmt --check` + `cargo clippy -- -D warnings` + 单元 + 媒体管线烟测 + 镜像 + SBOM |
| web | type check + eslint + 单元 + Playwright E2E（关键路径）+ 构建 + 视觉回归（可选） |
| app | analyze + 单元 + Widget 测试 + Patrol/Maestro E2E（关键路径）+ 构建 iOS/Android 产物 |
| device-firmware | 编译 + 静态扫描 + 单机自动化测试（USB 投喂机模拟器）+ OTA 包签名 |
| infra | terraform fmt/validate + plan diff comment |

每仓必须强制：

- 提交信息 Conventional Commits；CHANGELOG 自动生成。
- 必跑安全扫描：依赖漏扫（Snyk/Trivy）、密钥扫描（gitleaks）。

### 7.2 CD 流水线

- 服务镜像推送到内部 registry → ArgoCD/Flux 部署到 K8s；按 `env=dev/staging/canary/prod` 分目录。
- 灰度策略：见 05 第 10 节，结合 Argo Rollouts/Flagger 实现自动指标判定与回滚。
- 客户端：
  - Web：每次 `release/*` 触发预发，验证后切换 CDN 流量（蓝绿）。
  - App：CI 出包后上传 TestFlight / Firebase App Distribution / 内测平台；正式发版手动审核。
- 设备 OTA：先内部房间 → 1 家机构 → 10% 设备 → 100%，每步监控异常率。

### 7.3 环境

| 环境 | 用途 | 数据 |
| --- | --- | --- |
| dev | 开发自测 | mock + 公共测试设备 |
| staging | 预发，与 prod 同构（缩小规格） | 脱敏快照 |
| canary | 生产灰度 | 真实数据，1–5% 流量 |
| prod | 生产 | 真实数据 |

禁止把生产数据未脱敏导入 dev/staging。

## 8. 多端协作流程

### 8.1 新需求落地步骤

1. 产品在 `yunmao-platform` issue 描述需求 + 验收。
2. 架构/Tech Lead 评估是否涉及契约改动；若涉及，先开 `yunmao-contracts` PR，达成共识后合并并生成 SDK。
3. 各端基于新 SDK 版本独立开发；后端先发预发，端按 SDK 接入测试。
4. 多端集成测试由 QA 在 staging 完成（参见 09）。
5. 灰度发布，按数据决策放量。
6. 复盘 + 文档（00–11）更新。

### 8.2 跨端责任矩阵（RACI 节选）

| 工作项 | 产品 | 后端 | Web | App | QA | Ops |
| --- | --- | --- | --- | --- | --- | --- |
| 需求与 PRD | R | C | C | C | C | I |
| 契约设计 | C | R | C | C | I | I |
| 数据模型 | I | R | I | I | C | I |
| UI 实现 | C | I | R | R | C | I |
| 自动化测试 | I | R | R | R | R | I |
| 发布 | I | R | R | R | C | A |
| 应急 | I | R | R | R | R | A |

（R: Responsible, A: Accountable, C: Consulted, I: Informed）

### 8.3 文档治理

- 本目录 `docs/finalproductplanning/` 的 00–11 是规划层；契约规范在 `yunmao-contracts/`；服务设计文档在各服务仓 `docs/`。
- 任何破坏性变更必须同步更新对应文档；规划层文档由产品/架构 review，契约层由架构/相关业务 review。
- ADR 流程见 07 第 6 节。
- 文档评审使用 PR 评论 + 必要的语音会同步；最终通过的版本必须合并到主分支。

## 9. 跨语言协作要点

- gRPC：Go 与 Rust 服务统一用 Tonic/grpc-go；proto 文件在 contracts 仓维护；不允许在服务仓 fork proto。
- Kafka：Schema 在 contracts 仓维护，CI 校验向前兼容；Rust/Go 自动生成 codec。
- 日志/Trace：统一 OTel SDK 与 trace_id 传递格式（包括 MQTT/Kafka header）。
- 错误码：服务端返回 contracts 中定义的 `code`，端侧本地映射文案；不允许端硬编码文案。

## 10. 移动端发布特别约束

- iOS：虚拟权益必须 IAP；活动条款不能在 App 内提及外部支付。
- 应用商店审核材料统一由专人维护；首发优先 TestFlight + Android 内测渠道，避免审核被打回延误工期。
- 隐私合规：权限弹窗、SDK 清单、应用合规自检按各地（中国 + 国际）应用商店要求维护，相关材料挂在 `yunmao-platform/compliance/`。

## 11. 性能与体积预算（端内）

| 端 | 预算 |
| --- | --- |
| Web 首屏 | 关键 JS ≤ 200KB gz；首屏 LCP ≤ 2s（4G） |
| Web 房间页 | JS ≤ 350KB gz；播放器懒加载 |
| Flutter App | 包体 ≤ 35MB（不含动态资源） |
| 冷启动 (App) | iOS/Android ≤ 2s P95 |
| 关键页面 FPS | ≥ 55 |

预算变更需在 PR 中说明，并经性能负责人审阅。
